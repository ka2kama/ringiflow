# 技術記事の認知最適化

## 概要

長文の技術記事を「人間の認知に最適化」して設計するための知見をまとめる。認知心理学、情報アーキテクチャ、NN/g のアイトラッキング研究などを出典とする。

このドキュメントは記事の**構造設計**と**文章レベルの読みやすさ**の両方を扱う。

## 認知心理学の原則

技術記事の構造設計に直接適用できる認知心理学の原則。

### Progressive Disclosure（段階的開示）

読者が「今必要な情報」だけを提示し、詳細は要求に応じて開示する。

技術記事では UI のように動的に隠すことはできないが、**視覚的階層**で擬似的に実現する:

| レベル | 開示する内容 | 手法 |
|--------|------------|------|
| L1 | 記事タイトル + リード文 | 冒頭30秒で判断可能にする |
| L2 | 目次 + セクション冒頭要約 | 目次のスキャンで全体構造を把握 |
| L3 | 本文 | セクションに入って初めて読む |
| L4 | 補足・脚注・外部リンク | 折り畳み（`:::details`）、別ページ |

出典: [Progressive Disclosure - Nielsen Norman Group](https://www.nngroup.com/articles/progressive-disclosure/)

### Advance Organizer（先行オーガナイザー）

Ausubel の理論。新しい知識を既存の知識構造（スキーマ）に接続するための「足場」を最初に提供する。

| 種類 | 用途 | 記事での例 |
|------|------|----------|
| Expository（説明型） | 読者にとって未知のトピック | 「この記事で学ぶこと」リスト |
| Comparative（比較型） | 既知トピックとの関連がある | 「X を知っている人なら Y はこう理解できる」 |

単なる「目次」とは異なる。目次は構造を示すだけだが、Advance Organizer は**読者の既存知識との接続点**を明示する。

実装パターン:

```markdown
### この記事で得られること

- [既知の問題] に対する [新しいアプローチ] の方法
- [具体的な成果物] N 個
- [読者が持ち帰れるもの] の具体例
```

出典: [Advance Organizer - EduTech Wiki](https://edutechwiki.unige.ch/en/Advance_Organizer)

### チャンキングと Miller の 7±2

人間のワーキングメモリは一度に 5-9 個のチャンク（意味のまとまり）しか保持できない。

| 対象 | 推奨チャンク数 | 理由 |
|------|-------------|------|
| 記事全体の大セクション数 | 5-7 | 記事の全体像を把握するため |
| 各セクション内の小セクション数 | 3-5 | 1セクション内を通読するため |
| リスト項目数 | 5-9 | それ以上は分割するかグループ化 |
| テーブルの列数 | 3-5 | 横方向のスキャンの限界 |

項目数がこの範囲を超える場合は**グループ化**でメタレベルのチャンク数を収める。

出典: [Miller's Law | Laws of UX](https://lawsofux.com/millers-law/)

### Schema Theory（スキーマ理論）

人は新しい情報を、既存の知識構造（スキーマ）に同化または調節して理解する。

技術記事での適用:

1. **読者のスキーマを推定する**: 対象読者がどんな知識を持っているか明確にする
2. **既知から未知へ橋渡しする**: 新概念を導入するとき、まず読者が知っている概念との関係を示す
3. **アナロジーを活用する**: 「X は Y のようなもの」というメタファーは、既存スキーマを借りて新しいスキーマを構築する効率的な方法

カタログ形式の記事では、**統一テンプレート**がスキーマ構築に貢献する。読者は最初の2-3項目でセクションの構造パターンを学習し、以降は効率的にスキャンできる。

出典: [Schema Theory - Education Corner](https://www.educationcorner.com/schema-theory/)

### Dual Coding Theory（二重符号化理論）

Paivio の理論。人間は言語と非言語（視覚）の2チャネルで情報を処理する。両方を同時に活性化すると、記憶と理解が向上する。

| テキストの種類 | 対になるビジュアル |
|-------------|----------------|
| プロセス説明 | フローチャート、シーケンス図 |
| 概念の関係性 | 構造図、クラス図 |
| 比較・対照 | テーブル、マトリクス |
| 具体例 | コードブロック + 出力結果 |
| 時系列 | タイムライン |

**Split-Attention Effect への注意**: テキストとビジュアルが物理的に離れていると、読者の視線が行き来して認知負荷が増大する。図はそれを参照するテキストの**直後**に配置する。

出典: [Dual-coding theory - Wikipedia](https://en.wikipedia.org/wiki/Dual-coding_theory)

### Worked Example Effect（事例効果）

学習者に問題を自力で解かせるより、完全に解かれた例を見せる方が学習効率が高い。

技術記事での適用:

- 原則を述べた後、必ず具体例を示す
- 同じパターンの例が複数ある場合、最初を完全に示し、以降は段階的に省略する（Fading Effect）
- 対象読者が熟練者の場合は、完全な例がかえって学習を妨げることがある（Expertise Reversal Effect）

出典: [Worked-example effect - Wikipedia](https://en.wikipedia.org/wiki/Worked-example_effect)

### Concrete-before-Abstract（具体から抽象へ）

Bruner の CPA（Concrete-Pictorial-Abstract）アプローチ。具体的な例から始め、パターンを抽出し、最後に抽象的な原則にまとめる。

```
構造: 具体例 → パターンの抽出 → 抽象原則 → 他の具体例への適用
```

例:

1. 「この PR で実際に起きた品質問題」（具体事例）
2. 「共通するパターン: チェックリストの欠如」（パターン抽出）
3. 「品質ゲートの原則: 成果物には必ず検証ステップを設ける」（抽象原則）
4. 「この原則をドキュメントレビューに適用すると...」（他の文脈への転用）

ただし、読者が既にその分野のスキーマを持っている場合、Abstract-first（原則→具体例）の方が効率的なことがある。対象読者の熟練度による。

出典: [Concrete Examples Enhance Abstract Concepts - SAGE](https://journals.sagepub.com/doi/10.1177/00986283211058069)

## スキャナビリティの設計

NN/g のアイトラッキング研究から得られた、読者のスキャンパターンへの対応。

### Layer-Cake パターン（最重要）

読者は見出しだけを縦にスキャンする。**最も望ましいスキャンパターン**。

設計指針:

1. **見出しだけ読んで記事の要旨がわかるようにする**
2. 見出しは抽象的にせず、内容を端的に要約する
3. 見出し間の本文がスクロール 2-3 画面を超えない

| 見出しの例 | 評価 |
|-----------|------|
| 「概要」「基盤」「見えてきたこと」 | NG: 抽象的。内容が推測できない |
| 「品質を作り込む開発フロー」「失敗からプロセスを磨く改善サイクル」 | OK: セクション内容を要約している |

出典: [The Layer-Cake Pattern of Scanning - NN/g](https://www.nngroup.com/articles/layer-cake-pattern-scanning/)

### 視覚的階層

| 手法 | 用途 | 注意点 |
|------|------|--------|
| 見出しの階層（H2/H3/H4） | 構造の可視化 | H4 以降は避ける |
| 太字 | 文中の重要語を強調 | 乱用すると効果が消失 |
| 箇条書き | 並列情報の整理 | 文章として読ませたい場合は段落が適切 |
| テーブル | 比較・対照・マトリクス | 列数 3-5 が目安 |
| コードブロック | 実装例 | |
| Callout | 特に重要な情報、警告 | 役割（Info/Warning/Tip）を明確に |
| 図・ダイアグラム | プロセス、関係性の可視化 | Dual Coding Theory の適用 |

テーブルの多用に注意。**すべてがテーブルだと何も目立たなくなる**。テーブルで見せるべきもの（比較、分類）と散文で語るべきもの（ストーリー、洞察）を区別する。

出典: [Formatting Long-Form Content - NN/g](https://www.nngroup.com/articles/formatting-long-form-content/)

### Signposting（道標）

長文記事では読者が「今どこにいるか」を見失う。

| 道標の種類 | 実装 | 効果 |
|-----------|------|------|
| ロードマップ | 記事冒頭の「この記事は N パートで構成される」 | 全体の規模感と構造を事前把握 |
| 遷移文 | セクション末尾の「ここまでで X を見た。次は Y を見る」 | セクション間の接続 |
| 内部リンク | 「(詳細は [セクション名](#anchor) を参照)」 | 非線形な参照を可能にする |
| セクション冒頭の要約 | blockquote や1行サマリー | セクション単位の独立理解を支援 |

## 文章レベルの読みやすさ

構造設計だけでは記事は読みやすくならない。一つ一つのセクション・段落・文が読みやすくないと、構造がいくら良くても読者は離脱する。

### 一文を短く

- 修飾節が多いと、述語にたどり着くまでに疲れる
- 1文に入れる情報は1つ
- 文が長くなったら2文に分割する

### 一段落に詰め込まない

- 1段落で新出概念は1つまで
- 新出用語が3つ以上一気に出てくると読者が消化できない
- 段落が4行を超えたら分割を検討する

### 散文 vs. 箇条書きの判断基準

**箇条書きにすべきもの:**

- 並列する項目の列挙
- チェック項目
- 手順（番号付きリスト）
- 定義の一覧（太字のラベル + 説明）

**散文で書くべきもの:**

- 因果関係の説明（「A なので B」）
- 洞察や気づき
- 文脈の説明

**判定テスト:** 文中に「A、B、C など」や「X と Y と Z の3つがある」が出てきたら、箇条書きに分解できないか検討する。

### テーブルの使いどころ

テーブルは比較・対照・分類に使う。それ以外の用途（定義の列挙、チェックリスト）は箇条書きの方が読みやすい。

テーブルのセルに長文が入るのは、テーブルが適切なフォーマットではないサイン。

### 図表は道具であって装飾ではない

図表・箇条書きは読者の理解を助けるための道具。「ここに概念があるから図にしよう」ではなく「この概念はテキストだけでは伝わりにくいから図で補強しよう」で判断する。

**図を使うべき場面:**

- ループやサイクル（テキストで表現しにくい）
- 分岐のある判断フロー
- 階層構造
- 全体の中での位置づけ

**図が不要な場面:**

- 直後のテキストと完全に重複する
- ノードが3つ以下で、テキストで十分伝わる
- 読者がまだ知らない概念のラベルを並べただけ（全体像の図を冒頭に置くなど）
- 番号付きリストで済む線形なステップ

**判定テスト:** 図を削除してテキストだけ読んだとき、理解が落ちるか？ 落ちないなら図は不要。

### 見出しを惜しまない

見出しは読者がテキストブロックの内容を一瞬で把握するためのアンカー。見出しが足りないと、テキストが何についての文章なのかわからず目が滑る。

**見出しを追加すべきサイン:**

- 太字ラベル（`**守り:**` 等）で見出しを代用している
- 1つの H3 の中に複数の独立したトピックがある
- 重要な洞察が段落の中に埋もれている

太字ラベルを見出しにするか迷ったら、見出しにする。見出しが多すぎて困ることはまずない。

**冗長でも見出しをつける:** 人間は当然すぎる見出しは即スルーできるが、見出しがないテキストブロックは内容を読むまで役割がわからない。ナラティブな導入部であっても、ブロックに見出しがある方が認知負荷は低い。「順序で伝わるから見出しは不要」は書き手の錯覚であり、読者は順序通りに読むとは限らない。

**繰り返しの見出しは認知負荷を増やさない:** 各トピックに毎回「背景」「仕組み」のような汎用見出しがあっても、読者の認知負荷は増えない。むしろ予測可能なパターンの繰り返しは Schema Theory の観点でスキャン効率を高める。読者は最初の1-2項目でパターンを学習し、以降は見出し名だけで「読む/飛ばす」を即判断できる。Wikipedia や API ドキュメントが全エントリで同じ見出し構造を繰り返すのと同じ原理。

**カタログ形式では特に重要:** ストーリー形式なら文脈の流れが読者を導くが、カタログ形式では各概念がラベル付きで独立していないとスキャンできない。散文の中で概念を太字で導入するのではなく、見出しとして独立させる。

### 粒度は粗い→細かいの順で提示する

文章のあらゆるレベルで、粗い情報（カテゴリ、役割、定義）を先に出し、細かい情報（具体的内容、詳細）を後に出す。

| レベル | 粗い（先） | 細かい（後） |
|-------|----------|------------|
| 見出し | 役割プレフィックス（`背景:`） | 固有名（`なぜ3箇所か`） |
| セクション冒頭 | 1文定義（「〜する仕組みです」） | 背景・仕組みの詳細 |
| リスト/テーブルの前 | リード文（「以下の観点で確認します」） | リスト項目・テーブル行 |
| 段落 | トピックセンテンス | 具体的説明 |

粗いレベルが欠落すると、読者は細かい情報の**位置づけ**がわからない。「なぜ必要か」という見出しは具体的だが、それが背景なのか仕組みなのか補足なのかが一瞬で判別できない。`背景:` というプレフィックスがその判別を提供する。

**AI 生成文章に多い問題:** AI は文脈を既に知っているため、粗いレベルが自明に感じてスキップする。しかし読者はその文脈を持っていない。書き手にとっての冗長が、読者にとっての足場になる。

このセクション以下の原則（「これは何？」を先に、見出しを惜しまない）は、この原則の具体的な適用。

### 概念の導入には「これは何？」を先に

読者が概念の名前を見たとき、最初に知りたいのは「これは何？」であって「なぜ必要か？」や「どう使うか？」ではない。

- 見出し直後に1文の定義を置く（「〜するための仕組みです」「〜を分類した枠組みです」）
- 定義の後に、具体的な説明や必要性を展開する
- 定義がないまま背景説明や動機から始めると、読者は「で、結局何の話？」となる
- 定義文に構造的詳細（数量、内訳名）を詰め込まない。定義文は「何か」だけを伝え、構造は次の文に分離する

悪い例:

```markdown
### 設計ブラッシュアップ

設計の初版には必ず穴があります。穴がなくなるまでループします。
```

良い例:

```markdown
### 設計ブラッシュアップ

設計のギャップをゼロにするまで繰り返し改善するループです。

設計の初版には必ず穴があります。穴がなくなるまでループします。
```

定義文に構造的詳細を詰め込んだ悪い例:

```markdown
問題の表層ではなく根本原因にたどり着くための、6ステップの分析手順です。
```

良い例（定義と構造を分離）:

```markdown
問題の表層ではなく根本原因にたどり着くための分析手順です。全部で6ステップあります。
```

定義文は「何か」を伝える役割に専念し、「いくつあるか」「どんな内訳か」は次の文に分離する。

### 比喩の扱い

下手な比喩は使わないほうがマシ。比喩を使う場合:

- 読者全員が知っている対象に限定する
- 比喩なしでも意味が通じる文にする（比喩は補助）
- 比喩が本体の説明より長くならないこと

比喩に頼らず、具体例（実際に起きた事象）で説明する方が確実。

## Zenn/Qiita プラットフォームの読者特性

専門家向けベストプラクティス（NN/g 等）だけでは、Zenn/Qiita のカジュアルな読者層への最適化が不十分になる。

### 読者の行動モデル

- TOC から気になるセクションにジャンプする（通読しない）
- セクションの最初の2行で「読む価値があるか」を判断する
- 良いセクションだけを共有する（記事全体ではない）

### セクション冒頭の設計

- **共感フックで始める**: レポートサマリーではなく、読者の経験に響く問題を最初に置く
- **自己完結させる**: 他のセクションを読んだ前提の記述を避ける
- **数字への依存を避ける**: 「52件の分析の結果〜」ではなく、結論そのものを直接述べる

悪い例:

```markdown
52件の改善記録を分類した結果、対策には有効性の階層があることがわかりました。
```

良い例:

```markdown
AI がルールを守らないとき、どう対策しますか？
「ルールをもっと強調する」方向に行きがちですが、対策には明確な有効性の階層があります。
```

### セクション間の繋ぎ

フォーマルなシグナルポスティング（`> 次のセクションでは〜を紹介します`）はカジュアルなプラットフォームでは浮く。内容的な繋がりを自然に示す:

悪い例:

```markdown
> 次のセクションでは、この「自問」を体系化した問題解決フレームワークを紹介します。
```

良い例:

```markdown
この「対策の階層を自問する」習慣を、より体系的に実践するのが次の問題解決フレームワークです。
```

## 記事構造のパターン

### はじめにテンプレート

記事の導入部（はじめに）に必要な要素とその順序。読者は冒頭30秒で「読む価値があるか」を判断する（NN/g）。

| # | 要素 | 形式 | 役割 |
|---|------|------|------|
| 1 | コンテキスト + 問題提起 | H3 + 地の文 + callout | 読者との接点を作り、共感できる問題で引き込む |
| 2 | この記事の概要 + スコープ | H3 + 地の文 + callout | この記事が何を提供するか。用語・前提の定義 |
| 3 | 前提条件 | H3 | 記事が依存する前提知識や条件 |
| 4 | 得られること | H3 | Advance Organizer。本文セクションと対応 |

すべての要素に見出しを付ける。ナラティブな導入部であっても、見出しがあれば読者は一瞬で各ブロックの役割を把握できる。人間は当然すぎる見出しは即スルーできるが、見出しがないテキストブロックは内容を読むまで役割がわからない。

リード部分（1-2）は意味的に近い要素を統合し、4要素に整理する。見出し名は内容を端的に示す固有名にする（「コンテキスト」「概要」等の汎用名ではなく）。

**順序の根拠:**

- コンテキスト + 問題提起（1）: まず「何の話か」を示し、「何が問題か」で引き込む（SCR: Situation → Complication）
- 概要 + スコープ（2）: 「この記事で何がわかるか」と「前提を揃える」をセットにする
- 前提（3）→ 得られること（4）: 前提を共有した上で Advance Organizer を提示。読者はここで「自分に関係あるセクション」を判断して TOC にジャンプする

**得られることリストの設計:**

- 本文セクションと 1:1 で対応させる（`→ セクション N` で参照）
- 番号付きリストで記事全体の規模感を伝える
- 各項目は「読者が持ち帰れるもの」を述べる（記事の構成ではなく価値）

### 逆ピラミッド vs. 論理的進行

| パターン | 構造 | 適する記事 |
|---------|------|-----------|
| 逆ピラミッド | 結論 → 詳細 → 補足 | リファレンス、カタログ、FAQ |
| 論理的進行 | 前提 → 展開 → 結論 | チュートリアル、設計論 |
| ハイブリッド | 結論（要約）→ 論理的進行 → まとめ | 長文の技術解説記事 |

ハイブリッドのポイント: 冒頭で「結論（何が得られるか）」を示した上で、本文は論理的に展開する。スキャナビリティ（逆ピラミッド）と理解の深さ（論理的進行）を両立する。

### セクション独立性 vs. ナラティブフロー

長文記事には2つの読み方がある:

| 読み方 | 読者の動機 | 構造要件 |
|--------|----------|---------|
| 通読 | テーマに深い関心、体系的に学びたい | ナラティブフロー（論理的なつながり） |
| ピックアップ | 特定のセクションだけ読みたい | セクション独立性（前提知識の自己完結） |

両方を同時に満たすには:

- セクション冒頭に**1-2行のコンテキスト文**を置く（通読者には接続、ピックアップ読者には前提の明示）
- セクション末尾に**遷移文**を置く（通読者には区切り、ピックアップ読者には「このセクションで十分か」の判断材料）
- 依存関係が強いセクション間には**明示的なリンク**を張る

### カタログ形式固有の設計原則

Diataxis フレームワークで言えば、カタログ形式は **Explanation**（解説）に最も近い。

| 特性 | チュートリアル/ナラティブ | カタログ/リファレンス |
|------|-------------------------|---------------------|
| 読み方 | 順序通りに通読 | 必要な項目だけピックアップ |
| セクション間の依存 | 強い | 弱い（各項目が自己完結） |
| ナラティブの役割 | 物語の推進力 | カテゴリ間の意味的つながり |

カタログ形式で一貫性を保つための原則:

1. **統一テンプレート**: 各項目を同じ構造で記述する（スキーマ構築の効率化）
2. **冒頭のナビゲーションマップ**: カタログ全体の構造を図または表で示す
3. **クロスリファレンス**: 項目間の関係を明示的にリンクする
4. **カテゴリのグループ化**: トップレベルのカテゴリは 5-7 個以内（Miller の法則）

### トピック説明フレームワーク

カタログ形式の記事で各トピックを説明する際の統一テンプレート。読者の認知プロセス（「これは何？」→「なぜ必要？」→「どう動く？」→「実際にどう？」）に沿って要素を配置する。

#### 要素と順序

| # | 要素 | 必須/任意 | 見出し | 内容 |
|---|------|---------|--------|------|
| 1 | 定義 | 必須 | なし（H3 直後の地の文） | 1文で「これは何か」を述べる |
| 2 | 背景/問題 | 必須 | H4（内容が3行以上なら） | なぜこの仕組みが必要になったか |
| 3 | 仕組み/内容 | 概念による | H4 | どう動くか。ステップ、構成要素、図 |
| 4 | 具体例 | 推奨 | H4 | 実際に起きた事例 |
| 5 | 補足 | 任意 | H4 | なぜ機能するか、判定テスト、注意点 |

#### 見出し名の方針

**推奨: `汎用プレフィックス: 固有名` 方式**（例: `#### 背景: なぜ必要か`）。

汎用プレフィックスが役割マーカーとして機能し（Schema Theory: 予測可能なパターン）、固有名が内容のヒントを提供する（Layer-Cake: 見出しだけで内容がわかる）。両方の利点を取れる。

| プレフィックス | 役割 |
|-------------|------|
| `背景:` | なぜこの仕組みが必要か |
| `仕組み:` | どう動くか、何で構成されるか |
| `具体例:` | 実際に起きた事例 |
| `補足:` | なぜ機能するか、注意点、発展的な話題 |

固有名なしの `#### 仕組み` も有効（内容が1つの場合）。プレフィックスの繰り返しは認知負荷を増やさない。

#### テンプレート

```markdown
### [トピック名]

[1文定義。「〜するための仕組みです」「〜を分類した枠組みです」]

#### 背景: [固有名]

[なぜこの仕組みが必要になったか。読者が共感できる問題。]

#### 仕組み: [固有名]

[どう動くか。図、箇条書き、ステップ。]

#### 具体例: [固有名]

[実際に起きた事例。]

#### 補足: [固有名]（任意）

[なぜ機能するか、判定テスト、注意点。]
```

#### 適用の粒度

すべての H3 にフルテンプレートを強制しない。トピックの重さで判断する:

- **主要概念**（15行以上）: フルテンプレート。各要素を H4 で分離
- **補足的トピック**（15行未満）: 定義 + 背景 + 内容を地の文で。H4 細分化は不要
- **H2 レベル**: セクション全体の定義 + 導入。個別トピックは H3 で

出典: [Diataxis](https://diataxis.fr/)

## 内部ドキュメントと技術記事の違い

プロジェクト内部のドキュメントを外部向け技術記事に転換する際に注意すべき点。

| 観点 | 内部ドキュメント | 外部向け技術記事 |
|------|---------------|----------------|
| 読者 | プロジェクトの文脈を共有している | プロジェクトを知らない |
| 目的 | 正確な手順・仕様の記録 | 読者が自分の仕事に使える洞察の提供 |
| 詳細度 | 運用マニュアルレベルの詳細 | 原則と洞察。詳細は代表例で示す |
| 例 | プロジェクト固有のアーティファクト名 | 読者が共感できる普遍的な場面 |
| テーブル | チェックリスト項目の網羅的列挙 | 原則の比較・対照に限定 |

削る判断基準: **「この詳細は、プロジェクトを知らない読者が原則を理解するために必要か？」** — 必要なら残す。内部運用マニュアルに相当するなら原則の説明に置き換える。

具体例:

| 内部ドキュメント | 技術記事で書くこと |
|---------------|-----------------|
| 品質ゲートの8項目チェックリスト | 「守り（欠陥除去）と攻め（設計改善）の二面がある」という洞察 |
| PR完了フローの8ステップ | 「成果物の存在確認とユーザー承認で構造的に固める」という原則 |
| 計画ファイルの必須セクション一覧 | 「成果物要件として計画ファイルに必須要素を設ける」という手法 |

## 適用事例: AI 品質管理記事の構造改善

このナレッジを初めて適用した事例の記録。

### ギャップ分析で検出した問題

| ギャップ | 観点 | 対応 |
|---------|------|------|
| 「対策の3階層」が「失敗パターン」より先にある | Concrete-before-Abstract 違反 | 失敗パターンを前に移動し、3階層を帰納的結論として配置 |
| 「土台」セクションが15行で薄い | チャンキング | 「はじめに」に統合 |
| 「開発フロー」が295行・サブ6個 | 認知負荷 | 内部詳細を削減 |
| YAGNI/KISS が「開発フロー」内に混在 | 責務の単一性 | 開発フロー内に残すが、独立サブセクションとして明確化 |
| Advance Organizer がない | スキーマ接続 | 「この記事で得られること」リストを追加 |
| 見出しが抽象的（「土台」「基盤」） | Layer-Cake 違反 | 内容を要約する見出しに変更 |
| セクション冒頭の構造が不統一 | テンプレート不在 | blockquote による冒頭要約を統一 |
| セクション間の遷移文がない | Signposting | 主要セクション末尾に遷移文を追加 |
| 品質ゲート8項目等の内部詳細 | 内部ドキュメント的 | 原則の散文説明に置き換え |
| テーブルが40個以上 | 視覚的単調さ | テーブルと散文のバランスを調整 |

### v5 → v6: 文章レベルの読みやすさ

v5 は構造改善で読みやすくなったが、一つ一つのセクションが依然として読みづらかった。

検出した問題:

- 比喩が不適切（「ドアの閉め忘れ」「医者のたとえ」等）→ 全削除、具体例で説明
- テーブルの多用（定義や手順もテーブル）→ 箇条書き・番号付きリストに変換
- 一文が長い → 短文に分割
- 一段落に新出概念を詰め込みすぎ → 1概念ずつ
- レポート調のセクション冒頭 → 共感フックに変更
- 「52件の改善記録を分類した結果〜」等の他セクション依存 → 自己完結に

教訓: **構造設計（認知心理学フレームワーク）と文章レベルの craft は別の問題**。構造が良くても文が読みづらければ読者は離脱する。認知心理学の原則は必要だが十分ではない。

### v6 → v7: 見出しと概念定義

v6 は文章レベルで読みやすくなったが、「何の話をしているのかが一瞬でわからない」問題が残っていた。

検出した問題:

- 見出し（H4）が足りず、テキストブロックの目的が不明 → 12個以上の H4 見出しを追加
- カタログ形式なのに太字ラベルで見出しを代用 → 正式な見出しに昇格
- 概念に「これは何？」の定義がない → 見出し直後に1文定義を追加（8箇所）

教訓: **読者は概念名を見た瞬間に「これは何？」を知りたい**。背景や動機から入ると、概念の正体がわからないまま読み進めることになり、目が滑る。

### v7 → v8: テンプレート統一と連番

v7 まではセクションごとに個別改善を重ねてきたが、v8 で「再現可能なテンプレート」として体系化した。

適用した変更:

- **トピック説明フレームワーク**: 各 H3 トピックを「定義 → 背景 → 仕組み → 具体例 → 補足」の統一構造に restructure
- **はじめにテンプレート**: 導入部を4要素に整理し、すべてに H3 見出しを付与。ナラティブな導入部でも見出しがある方が認知負荷が低い
- **連番とグルーピング**: H2 セクションに 1-5 の連番。フローステップに 3-1〜3-5、改善サイクルに 4-1〜4-3。横断トピックは「コラム:」プレフィックス
- **「得られること」とセクションの対応**: 番号付きリスト + `（→ セクション N）` で Advance Organizer と本文を接続

教訓: **個別改善の蓄積がテンプレートの原材料になる**。最初からテンプレートを設計しようとしても、どの要素が必要かは実際に何度も書き直さないとわからない。

### 結果

892行（v4）→ 489行（v5）→ 491行（v6）→ 453行（v7）→ 490行（v8）。v8 は構造の統一が主な変更で、行数は連番・見出し追加分で微増。
