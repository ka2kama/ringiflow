# 実装解説

ユーザー操作を起点に、システム全体でどう処理が連携するかを解説するドキュメントを管理する。

> ⚠️ 本ディレクトリの内容は作成当時の解説であり、コード変更に追従してメンテナンスされるものではない。

## 原則

1. **E2E フローが主役**: ユーザー操作 → Frontend → BFF → Core Service → DB → レスポンス → 画面更新の流れを sequenceDiagram で可視化する
2. **コンポーネント間の連携を解説する**: API 契約、型変換、エラー伝播など、1つのファイルからは見えない境界部分に価値がある
3. **内部処理は書かない**: ファイル内部のロジックはコードを読めばわかるようにする。解説が必要ならコードの可読性を改善すべき

## 目的

| 読者の問い | 実装解説が答えること |
|-----------|-------------------|
| この操作でシステム全体に何が起きるか？ | E2E フロー（sequenceDiagram） |
| コンポーネント間でデータはどう変換されるか？ | 境界部分の型変換・API 契約 |
| なぜこの仕組みを採用したか？ | 設計判断（選択肢と採用理由） |
| この横断的関心事はどこで発動するか？ | 発動タイミングと仕組み |

## 生成方法

`/explain` で生成する。

```bash
/explain 123          # PR #123 の差分を解説
/explain 認証フロー    # 既存機能を解説
```

## sequenceDiagram の記法ルール

### participant の命名: 役割で統一

participant 名はアーキテクチャ上の役割で統一する。技術名（Elm, PostgreSQL 等）は使用しない。技術スタックは README や概要セクションで記載する。

| participant ID | ラベル | 備考 |
|---------------|--------|------|
| `U` | ユーザー | 必要に応じて「（承認者）」等の補足 |
| `FE` | Frontend | |
| `BFF` | BFF | |
| `Redis` | Redis | 技術名だが役割名（Session Store）より通じやすい |
| `Core` | Core Service | |
| `DB` | DB | |
| `Notify` | Notification | |

### self-message の禁止

`A->>A:` 形式の self-message は使用しない。self-message は実際の通信先を隠し、読者に暗黙知を要求する。

| 処理の性質 | 記法 | 例 |
|-----------|------|-----|
| 外部通信を伴う | 通信先を participant として矢印で表現 | `BFF->>Redis: セッション取得` |
| 外部通信を伴わない判断・計算 | `Note over` で表現 | `Note over UC: 権限チェック` |

### 1 つの図 = 1 つのユーザー操作

独立したタイミングで実行できるユーザー操作は、別の図に分ける。1 つの図に複数の操作をまとめると「連続した 1 回の操作」という誤った印象を与える。

操作間に前提条件がある場合は、図の前の説明文に明記する。

### ユーザー操作はコンパクトに

ブラウザ内で完結する UI 操作（ボタン押下、確認ダイアログ等）は 1 行にまとめる。E2E フローの主題はコンポーネント間通信であり、UI 内部のやり取りは詳細化しない。

```
// 良い: 1 行で操作の意図が伝わる
U->>FE: 「承認」ボタン → 確認ダイアログで確定

// 避ける: UI 内部の往復を描いている
U->>FE: 「承認」ボタン
FE-->>U: 確認ダイアログ表示
U->>FE: 「確認」ボタン
```

### 番号付きブロックと同期/非同期の明示

処理の流れを番号付きの `Note over` ブロックで区切り、各ブロックの性質（同期/非同期）を明示する。

```
Note over Core, DB: 3. 検証（同期）        ← 番号 + 性質
Core->>DB: SELECT ...
DB-->>Core: row                            ← 戻り矢印で同期を表現

Note over Core, Notify: 5. 通知（fire-and-forget）
Core-)Notify: Approved → 申請者            ← -) で非同期を表現
```

| 記法 | 意味 | 用途 |
|------|------|------|
| `A->>B` + `B-->>A` | 同期呼び出し（リクエスト → レスポンス） | DB クエリ、API 呼び出し |
| `A-)B` | 非同期（fire-and-forget、結果を待たない） | 通知送信 |
| `Note over A, B: N. 見出し（性質）` | ブロック区切り | 処理フェーズの境界 |

### ズームレベルの分離

1 枚の図にすべての粒度を詰め込まない。

| ズームレベル | 対象 | participant の粒度 |
|------------|------|-------------------|
| コンポーネント間 | E2E フロー | Frontend, BFF, Redis, Core Service, DB, Notification |
| レイヤー間 | コンポーネント内部 | Handler, UseCase, Domain, Repository, DB |

コンポーネント間の図を先に配置し、レイヤー間の図を補足セクションとして後に配置する。

## 文書の構成

### 機能ドメイン

```markdown
# <機能名>

対応 PR: #NNN, #NNN（出典）

## 概要
ユーザー視点で「何ができるか」を 1-2 文で。

## E2E フロー
sequenceDiagram でユーザー操作からレスポンスまでの全体像を示す。

### 正常系
（sequenceDiagram）

### 準正常系・異常系（必要に応じて）
エラー時のフロー。

## コンポーネント間の境界
API 契約、型変換、データ変換など「コードを読むだけでは見えない」部分。

## 状態遷移（該当する場合）

## 設計判断
仕組み・方式レベルの選択のみ。
```

### 横断的関心事

```markdown
# <横断的関心事名>

対応 PR: #NNN（出典）

## 概要

## いつ・どこで発動するか
具体的な機能フロー内での発動タイミング。

## 仕組み

## 設計判断
```

## 他のドキュメントとの関係

| ドキュメント | 役割 | 実装解説との違い |
|-------------|------|-----------------|
| 設計書 | 事前の設計仕様 | 実装解説は「実際にどう動いているか」の E2E フロー |
| ADR | 意思決定の記録 | 実装解説は決定の結果どう連携しているかを可視化 |
| ナレッジベース | 汎用的な技術知識 | 実装解説はプロジェクト固有のフロー |
| セッションログ | 作業記録 | 実装解説は体系的な解説 |
