# ADR-063: PR レビュープロンプトのベストプラクティス起点再設計

## ステータス

承認済み

## コンテキスト

`methodology-design.md` は「既知手法を地図として把握した上で設計する」ことを求めているが、PR レビュープロンプト（`claude-auto-review.yaml`、`claude-rules-check.yaml`）の設計にはこの原則が適用されていなかった。

直近 20 PR の claude[bot] レビューコメントを分析した結果、以下の問題が判明した:

1. Rules Check の全違反が Critical 扱い → ドキュメントフォーマット違反（太字使用等）で action-required: true になり、再レビューサイクルが発生
2. Verification の検出力が未証明 — 20 PR で Critical: 0、High: 0、Medium: 2（1 PR のみ）。セキュリティ/パフォーマンスは観点に含まれているが検出実績ゼロ
3. 再レビュー時に全 4 種類のレビューを再実行し、LGTM コメントが大量に残る

## 検討した選択肢

### 選択肢 1: 既知手法ベースの段階的改善（採用）

既知のコードレビューベストプラクティスを調査し、現行プロンプトを段階的に改善する。

評価:
- 利点: 理論的根拠に基づく改善、既存の強み（V&V 分離、計画突合精度）を維持
- 欠点: 改善効果の検証に時間がかかる

### 選択肢 2: Conventional Comments フォーマットの全面採用

AI コメントに `label (decorator): subject` フォーマットを全面導入する。

評価:
- 利点: コメントの意図が構造的に明確になる
- 欠点: AI コメントは既にメタデータ（`<!-- review-metadata -->`）で機械的に分類されており、テキストラベルは冗長。人間がコメントを読む際の可読性は向上するが、既存のメタデータ方式との二重管理になる

### 選択肢 3: 現状維持

評価:
- 利点: 変更リスクなし
- 欠点: 実測データが示す問題（重大度インフレ、alert fatigue）が継続

### 比較表

| 観点 | 選択肢 1 | 選択肢 2 | 選択肢 3 |
|------|---------|---------|---------|
| 理論的根拠 | 複数手法の統合 | 単一手法の採用 | なし |
| 既存の仕組みとの整合 | 高（段階的改善） | 低（フォーマット二重管理） | 高 |
| 問題の解決 | 部分的（段階的） | 部分的（フォーマットのみ） | 未解決 |

## 決定

選択肢 1: 既知手法ベースの段階的改善を採用する。

具体的な変更:

### 1. Rules Check の重大度モデル再設計

変更前: 全ルール違反が Critical（`severity-critical` にカウント、`action-required: true`）

変更後: ルール違反を 2 階層に分類

| 違反の種類 | 重大度 | blocking |
|-----------|--------|---------|
| 禁止事項（`**禁止:**` に該当する違反） | High | blocking |
| その他のルール違反（スタイル、推奨事項） | Medium | non-blocking |

根拠: Conventional Comments の blocking/non-blocking デコレータ。Google Code Review Guidelines の Nit: 概念。

### 2. Verification 観点の再構成

変更前の 6 観点を Google Code Review Guidelines を参照して 7 観点に再構成。AI コードレビュー研究に基づき、確信度の明示とコメント数のガイドラインを追加。

| # | 観点 | blocking 条件 | 根拠 |
|---|------|-------------|------|
| 1 | 正確性・ロジック | バグが確実な場合 | Google: Functionality |
| 2 | セキュリティ | 高確信度の脆弱性のみ | AI 研究: FPR 最小化 |
| 3 | パフォーマンス | 明白なボトルネックのみ | SmartBear: 確信度閾値 |
| 4 | 型安全性・堅牢性 | 型安全性の明確な欠如 | プロジェクト固有 |
| 5 | テストの妥当性 | テストの重大な欠落 | Google: Tests |
| 6 | 設計・複雑さ | 責務混在、依存違反 | Google: Design + Complexity |
| 7 | コード改善 | non-blocking 固定 | Bacchelli: 実際のレビュー成果で最多 |

### 3. レビュー効率改善

- Rules Check LGTM の簡略化: 1 行 + メタデータのみ（ルール一覧を省略）
- Verification に「コメント数ガイドライン」を追加: 重要な問題に絞る

## 帰結

### 肯定的な影響

- Rules Check の重大度インフレが解消され、ドキュメントスタイル修正のための再レビューサイクルが削減される
- Verification 観点が既知手法で裏付けられ、methodology-design.md の原則との整合が取れる
- LGTM コメントの簡略化で PR 上のノイズが削減される

### 否定的な影響・トレードオフ

- Rules Check の non-blocking 違反が放置されるリスク — ただし LGTM コメントではなくコメントとして残るため、実装者は認識できる
- Verification 観点の変更により、AI の指摘パターンが変わる可能性 — 効果は次回の `/retro` で検証する
