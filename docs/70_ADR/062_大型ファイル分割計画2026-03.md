# ADR-062: 大型ファイル分割計画（2026-03）

## ステータス

承認済み

## コンテキスト

健全性診断（2026-03）で、500行超ファイルが 16→34 に倍増、1000行超が 6 件に達していることが検出された（Issue #996）。

ADR-043 で確立した分析手法（prod/test 分離、例外許容基準）を踏襲し、新たに閾値を超過した 6 ファイルについて対応方針を確定する。

### ADR-043 との関係

ADR-043（2026-02-10）は Issue #290 で検出された大型ファイルを分析し、handler/workflow.rs の CQRS 分割と New.elm の ApproverSelector 抽出を決定した。本 ADR は ADR-043 のフォローアップとして、Feature 開発加速の副作用で新たに肥大化したファイルを対象とする。

分析手法は ADR-043 を踏襲する:
- prod/test 分離分析で実質的なプロダクションコード量を把握
- 例外許容基準: prod < 500 + 高凝集

## 決定

### 例外許容（2ファイル）

ADR-043 の例外許容基準（prod < 500 + 高凝集）に該当するため、分割不要と判断する。

| ファイル | prod行数 | 凝集度 | 理由 |
|---------|---------|--------|------|
| document.rs | ~306 | 極めて高い | ドキュメント S3 ライフサイクルに特化。6 エンドポイントが同一ドメイン |
| approve.rs | ~291 | 極めて高い | 単一の承認操作に集中。4 関数が密結合 |

### 分割対象（4ファイル）

#### 対象 1: Designer.elm（1965行 → 7ファイル）

分割方法: **View 層のコンポーネント抽出 + Update ロジックの分離**

当初は 6 ファイル構成を計画していたが、実装時に Elm の循環依存制約が判明し、共有型定義を独立モジュール `Types.elm` として追加した 7 ファイル構成とした（PR #1013）。Designer.elm が Msg を定義し、サブモジュールが Msg を参照する一方で、Designer.elm がサブモジュールの関数を呼び出す双方向依存は Elm では許されないため、Types.elm に共有型を集約して一方向の依存グラフを実現した。

| 新モジュール | 内容 | 実績行数 |
|-------------|------|---------|
| `Designer.elm`（本体） | init, subscriptions, update ルーティング, viewLoaded レイアウト | 165 |
| `Designer/Types.elm` | 共有型定義（Model, PageState, CanvasState, Msg） | 117 |
| `Designer/Update.elm` | updateLoaded の全ケース + ヘルパー関数 | 696 → 462 |
| `Designer/UpdatePersistence.elm` | Save/Validate/Publish チェーン + 接続線付け替え | 352 |
| `Designer/Canvas.elm` | SVG キャンバス描画（ステップノード、グリッド、ドラッグプレビュー） | 659 → 325 |
| `Designer/CanvasTransitions.elm` | 接続線 SVG 描画（矢印マーカー、ベジェ曲線、ドラッグプレビュー、付け替えハンドル） | 351 |
| `Designer/PropertyPanel.elm` | プロパティパネル（ステップ/接続線のプロパティ表示・編集） | 158 |
| `Designer/Palette.elm` | ステップ選択パレット + アイコン描画 | 90 |
| `Designer/Toolbar.elm` | ツールバー、メッセージ表示、バリデーション結果、ステータスバー、公開ダイアログ | 173 |

Issue #1019 で Update.elm と Canvas.elm をさらに分割し、全ファイルが 500行以下となった（PR #1021）。

Elm のモジュール構造:
```
Page/WorkflowDefinition/
├── Designer.elm          # 本体（init, update ルーティング, subscriptions, view レイアウト）
└── Designer/
    ├── Types.elm         # 共有型定義（循環依存回避）
    ├── Update.elm        # updateLoaded ロジック（ドラッグ、プロパティ編集、選択、キーボード）
    ├── UpdatePersistence.elm  # Save/Validate/Publish チェーン + 接続線付け替え
    ├── Canvas.elm        # SVG キャンバス描画（ステップノード、グリッド、ドラッグプレビュー）
    ├── CanvasTransitions.elm  # 接続線 SVG 描画
    ├── PropertyPanel.elm # プロパティパネル
    ├── Palette.elm       # ステップパレット
    └── Toolbar.elm       # ツールバー・ダイアログ
```

選択理由:
1. View 層の抽出は ApproverSelector で確立済みのコールバックパターンに従う
2. Update 層の分離は Elm の TEA パターンで一般的な手法（別モジュールで `impl` 相当の関数を公開）
3. ページ固有のサブモジュールは `Page/X/X/` ディレクトリに配置し、Component/ は再利用可能なもののみに限定
4. Types.elm パターンは Elm コミュニティの定番手法。コールバックレコードパターンや Msg の分散よりもボイラープレートが少ない

検討した代替案:
- Component/ への配置 → ページ固有の UI であり再利用性がないため不適切
- Nested TEA による完全分離 → 過度なボイラープレートが発生し、ADR-043 でも否定された
- コールバックレコードパターン → 循環依存は回避できるが、各 view にレコードを渡すボイラープレートが増加

#### 対象 2: Detail.elm（1479行 → 各ファイル ≤500行）

分割方法: **機能ドメイン別のモジュール分離**

3 つの独立した機能ドメイン（承認、再提出、コメント）をサブモジュールとして抽出する。

| 新モジュール | 内容 | 推定行数 |
|-------------|------|---------|
| `Detail.elm`（本体） | Model, init, Msg ルーティング, viewLoaded レイアウト, 基本表示（タイトル、ステータス、ステップ、基本情報） | ~450 |
| `Detail/Approval.elm` | 承認/却下/差し戻し操作 + 確認ダイアログ + コメント入力 | ~270 |
| `Detail/Resubmit.elm` | 編集モード + フォーム入力 + 承認者選択管理 + バリデーション | ~400 |
| `Detail/Comments.elm` | コメント一覧表示 + 投稿フォーム + 取得 | ~200 |
| `Detail/StepProgress.elm` | ステップ進捗バー（純粋 View） | ~90 |

Elm のモジュール構造:
```
Page/Workflow/
├── Detail.elm            # 本体
└── Detail/
    ├── Approval.elm      # 承認操作
    ├── Resubmit.elm      # 再提出・編集
    ├── Comments.elm      # コメント
    └── StepProgress.elm  # 進捗表示（純粋 View）
```

選択理由:
1. 承認・再提出・コメントは互いに独立した機能ドメイン（状態・メッセージ・UI が自己完結）
2. StepProgress は純粋 View（状態管理なし）で最も抽出が容易
3. LoadedState の 11 フィールドが 3 つのドメインに自然に分解される

#### 対象 3: instance.rs（prod ~754行 → 各ファイル ≤500行）

分割方法: **責務別のサブモジュール分離**

ADR-043 で確立済みの Rust モジュール分割パターン（parent.rs + sub-directory）を適用する。

| 新モジュール | 内容 | 推定行数（prod） |
|-------------|------|----------------|
| `instance.rs`（親モジュール） | WorkflowInstance 構造体, new(), アクセサ, re-exports | ~275 |
| `instance/state.rs` | WorkflowInstanceState, 各状態型, CancelledState + impl | ~120 |
| `instance/transitions.rs` | 8 つの状態遷移メソッド（impl WorkflowInstance） | ~230 |
| `instance/record.rs` | WorkflowInstanceRecord, from_db()（不変条件検証） | ~160 |

Rust のモジュール構造:
```
backend/crates/domain/src/workflow/
├── instance.rs           # 親モジュール（構造体定義 + re-exports）
└── instance/
    ├── state.rs          # 状態機械の型定義
    ├── transitions.rs    # 状態遷移メソッド
    └── record.rs         # DB レコード変換
```

選択理由:
1. handler/workflow.rs の分割パターン（parent.rs + sub-directory）と一貫
2. Rust の impl ブロック分割（同一クレート内で複数ファイルに分散可能）を活用
3. テストは各サブモジュール内の `#[cfg(test)]` に分散配置

#### 対象 4: New.elm（1099行 → 4ファイル）

分割方法: **ADR-043 の方針を拡張**

ADR-043 で ApproverSelector 抽出は完了済み。残りの責務を分離する。

当初は 3 ファイル構成を計画していたが、Designer.elm（対象 1）および Detail.elm（対象 2）と同様に、Elm の循環依存制約により共有型定義を独立モジュール `Types.elm` として追加した 4 ファイル構成とした（PR #1018）。

| 新モジュール | 内容 | 実績行数 |
|-------------|------|---------|
| `New.elm`（本体） | init, update ルーティング, updateShared, isDirty, view レイアウト | 221 |
| `New/Types.elm` | 共有型定義（Model, PageState, LoadedState, FormState, EditingState, Msg, initEditing） | 149 |
| `New/Update.elm` | updateLoaded, updateEditing + ヘルパー関数（バリデーション） | 540 → 447 |
| `New/Api.elm` | API 呼び出し（下書き保存、申請、保存+申請、フォーム値エンコード） | 69 |
| `New/FormView.elm` | フォーム入力 UI（定義選択、タイトル、動的フィールド、承認者、アクションボタン） | 259 |

Issue #1019 で DirtyState の共通化と API 関数の抽出により、Update.elm を 500行以下に縮小した（PR #1021）。

Elm のモジュール構造:
```
Page/Workflow/
├── New.elm               # 本体（init, update ルーティング, view レイアウト）
└── New/
    ├── Types.elm         # 共有型定義（循環依存回避）
    ├── Update.elm        # Update ロジック
    ├── Api.elm           # API 呼び出し
    └── FormView.elm      # フォーム表示
```

選択理由:
1. ADR-043 で Nested TEA の過度な階層化は否定済み。サブモジュール分離で十分
2. Update ロジックと View の分離は自然な境界
3. Types.elm パターンは Designer.elm、Detail.elm で確立済み。循環依存回避の定番手法
4. DefinitionSelector は ~44行と小さく、独立コンポーネント化する価値が低い

### 実施優先順位

| 順位 | ファイル | 行数 | 分割数 | 理由 |
|------|---------|------|--------|------|
| 1 | Designer.elm | 1965 | 7 | 最大のファイル、最多の責務混在（Types.elm 追加で循環依存回避） |
| 2 | Detail.elm | 1479 | 5 | 独立した機能ドメインの混在が顕著 |
| 3 | instance.rs | 1392 (prod 754) | 4 | ドメイン層の中核、状態機械の明確な分割軸あり |

## 帰結

### 肯定的な影響

- 6 ファイル中 4 ファイルを分割し、1000行超ファイルを 0 にできる
- 各サブモジュールが単一責務となり、変更時の影響範囲が明確になる
- ADR-043 で確立した分割パターン（Rust: parent.rs + sub-directory、Elm: コールバックパターン）を一貫して適用

### 否定的な影響・トレードオフ

- Elm のサブモジュール分離は import パスが長くなる（`Page.WorkflowDefinition.Designer.Canvas`）
- Update ロジックの分離はメッセージルーティングのボイラープレートが増加する
- 分割後のファイル間依存関係の管理が必要（特に Elm の型共有）
