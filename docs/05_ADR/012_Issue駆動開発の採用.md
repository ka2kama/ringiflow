# ADR-012: Issue 駆動開発の採用

## ステータス

承認済み

## コンテキスト

RingiFlow プロジェクトでは、タスク管理と進捗の可視化が必要である。
このプロジェクトには以下の特徴がある:

- 個人開発 + AI エージェント（Claude Code）との協働
- 学習目的（プロジェクト管理手法自体も学習対象）
- 暗黙知ゼロの方針（すべてを文書化）

タスク管理の方法を決定する必要がある。

## 検討した選択肢

### 選択肢 1: セッションログのみ

会話ベースで進め、セッションログに記録する。

評価:
- 利点: オーバーヘッドが少ない、柔軟
- 欠点: 全体像が見えにくい、「何が残っているか」が分かりにくい

### 選択肢 2: GitHub Projects + Issue 駆動

Issue でタスクを管理し、PR と紐づける。Project Board で可視化する。

評価:
- 利点: 業界標準のプラクティス、履歴が残る、学習になる
- 欠点: Issue 作成・管理のオーバーヘッド

### 選択肢 3: 外部ツール（Notion、Jira など）

GitHub 外のツールでタスク管理を行う。

評価:
- 利点: 高機能、柔軟なカスタマイズ
- 欠点: GitHub との連携が手間、ツールが分散する

### 比較表

| 観点 | セッションログのみ | GitHub Projects | 外部ツール |
|------|-------------------|-----------------|-----------|
| オーバーヘッド | 低 | 中 | 高 |
| 可視性 | 低 | 高 | 高 |
| 学習効果 | 低 | 高 | 中 |
| GitHub 統合 | - | ネイティブ | 要連携 |

## 決定

**GitHub Projects + Issue 駆動開発を採用する。**

理由:
1. プロジェクト管理手法自体が学習対象であり、業界標準のプラクティスを学べる
2. Issue → ブランチ → PR → マージ のフローが自然に身につく
3. 暗黙知ゼロの方針と整合する（すべてが GitHub 上に記録される）

## 帰結

### 肯定的な影響

- タスクの可視化と進捗管理が可能になる
- PR と Issue の紐づけにより、変更の理由が追跡可能になる
- Milestone で Phase ごとの進捗が明確になる
- 実際のチーム開発と同様のフローを経験できる

### 否定的な影響・トレードオフ

- Issue 作成・管理のオーバーヘッドが発生する
- 小さなタスクまで Issue 化すると煩雑になる可能性がある
  - 対策: 大きな機能単位で Issue を作成し、細かいタスクは Issue 内のチェックリストで管理
