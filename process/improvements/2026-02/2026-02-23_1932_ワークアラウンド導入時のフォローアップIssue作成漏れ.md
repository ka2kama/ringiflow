# ワークアラウンド導入時のフォローアップ Issue 作成漏れ

## 事象

PR #825（match-rules.py → match-rules.rs 移行）で、`workflow_run` トリガーの制約により削除した match-rules.py を一時的に復元するワークアラウンドを導入した。コミットメッセージにはマージ後に削除予定と記載したが、クリーンアップ用のフォローアップ Issue を作成しなかった。マージ完了後にユーザーから指摘を受け、その際もユーザーに検討を委ねた。

## 原因分析

### Want

ワークアラウンド導入時に、クリーンアップが確実に追跡・実行されること。

### To-Be

ワークアラウンドを導入した時点で、クリーンアップ用のフォローアップ Issue が作成され、追跡可能な状態になっている。

### As-Is

1. ワークアラウンド導入のコミットメッセージに「マージ後に削除予定」と記載した
2. フォローアップ Issue を作成しなかった
3. マージ完了後、ユーザーにクリーンアップの判断を委ねた

### ギャップ分析

- コミットメッセージをタスク追跡の手段として扱った。コミットメッセージはナレッジであり、タスク追跡は GitHub Issues に一元化する方針（ADR-025）に反する
- ワークアラウンドの導入を「情報提供」と位置づけ、クリーンアップの追跡を自分の責務として認識しなかった

### 根本原因

ワークアラウンド導入時にフォローアップ Issue を作成するプロセスが定義されていない。zoom-rhythm.md の「暫定値・ワークアラウンド」セクションは影響パスの洗い出しのみをカバーしており、クリーンアップ追跡の要件がない。

## 対策

問題の性質: プロセス的（ワークアラウンド導入時のフロー不備）

### 対策案: ワークアラウンド導入時のフォローアップ Issue 作成を成果物要件化

zoom-rhythm.md の「暫定値・ワークアラウンド」セクションを拡張し、クリーンアップ用フォローアップ Issue の作成を成果物要件として追加する。

成果物要件: ワークアラウンドを導入するコミットまたは PR に、対応するフォローアップ Issue 番号が記載されていること。

| 対策形式 | 案 | 採用 |
|---------|-----|------|
| 行動規範 | 「ワークアラウンド導入時は Issue を作成すること」 | 不採用（再発リスク高） |
| 成果物要件 | フォローアップ Issue の存在と番号記載を要件化 | 採用 |
| フロー組み込み | lefthook でコミットメッセージを自動検証 | 不採用（ワークアラウンドの判定が自動化困難） |

配置先: `.claude/rules/zoom-rhythm.md`（暫定値・ワークアラウンドセクション）
