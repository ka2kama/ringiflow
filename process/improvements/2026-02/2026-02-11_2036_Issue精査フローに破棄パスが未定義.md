# Issue 精査フローに破棄パスが未定義

## 事象

Issue 駆動開発フローの「既存 Issue の精査」セクションでは、5つの観点（Want の確認、How への偏り、完了基準の妥当性、暗黙の仮定、スコープの適切さ）で Issue を検証するプロセスが定義されている。

しかし、精査結果の分岐が「修正して続行」の一択になっている:

> 精査の結果、Issue の修正が必要な場合は修正してから次のステップに進む。

「Issue は仮説であり精査対象」と前提に明記しながら、仮説が棄却される経路（Issue の破棄・クローズ）が定義されていない。

## 原因分析

### 直接原因

Issue 精査プロセスの追加時（[Issue を正として扱う暗黙の前提](./2026-02-06_1946_Issueを正として扱う暗黙の前提.md) の対策として）、精査結果のパスとして「破棄」を考慮しなかった。

### 根本原因

精査プロセスの設計が「Issue の内容を改善する」方向にのみフォーカスしており、「Issue 自体の必要性を判断する」視点が欠落していた。仮説検証は棄却を含めて初めて成立するが、棄却パスのない検証プロセスは確証バイアスを構造的に許容する。

破棄が妥当なケース:

| ケース | 例 |
|--------|-----|
| Want が存在しない | 前提にしていた問題が再現しない、環境変化で解消済み |
| Want は別の Issue で充足 | 重複、または別 Issue の一部として吸収 |
| 前提が根本的に誤り | Issue 全体の前提崩壊、修正では対応不能 |
| コスト対効果が見合わない | Want は実在するが、現時点の優先度では見合わない |

## 対策

精査結果の分岐を複数パスに拡張する。Issue 駆動開発フローの「既存 Issue の精査」セクションに、精査結果の分岐表を追加する:

| 精査結果 | 条件 | アクション |
|---------|------|-----------|
| 続行 | Issue の内容に問題なし | ブランチ作成へ |
| 修正して続行 | スコープ・完了基準等に修正が必要 | Issue を修正後、ブランチ作成へ |
| 再構成 | 分割・統合・再作成が必要 | 新 Issue を作成し、元 Issue をクローズ |
| 破棄 | Want 不在、前提崩壊等 | 理由を記録して Issue をクローズ |

対策形式: 成果物要件（精査結果の分岐表が精査セクションに存在すること）
