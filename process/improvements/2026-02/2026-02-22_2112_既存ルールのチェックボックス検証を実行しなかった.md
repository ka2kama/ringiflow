# 既存ルールのチェックボックス検証を実行しなかった

## 事象

Issue #778 のマージ後、完了基準のチェックボックス 6 項目を検証せずに一括チェックした。`.claude/rules/docs.md` の自己チェック #5「チェックボックスは検証完了後のみチェックする。検証せずにチェックしてはならない」に違反している。

指摘を受けて事後検証を実施し、全 6 項目の充足をファイル確認・CI 結果で確認した。結果的に全項目 OK だったが、検証プロセスの省略自体が問題。

## 原因分析

### 表層原因

Issue クローズ済み + PR マージ済みの状態で、チェックボックスの更新を「残作業の消化」として扱い、検証ステップを省略した。

### 根本原因

ルールは存在するが（docs.md #5）、Issue のチェックボックス更新という具体的な操作フローの中で参照されていない。docs.md の自己チェックは「AI がドキュメントを生成する際」のチェックリストであり、`gh issue edit` で Issue 本文を更新する操作フローには組み込まれていない。

## 対策

問題の性質: プロセス的（既存ルールが操作フローに組み込まれていない）

### 即時対策（実施済み）

事後検証を実施し、6 項目すべてのエビデンスを確認した。

### 恒久対策

Issue のチェックボックス更新は wrap-up または review-and-merge の中で行われる。これらのスキル定義に検証ステップを明示する方法が考えられるが、対策形式の検討が必要。

対策形式の検討:
- 行動規範（「検証してからチェックする」）: docs.md #5 に既に存在するが守られなかった。同じ形式を別の場所に追加しても効果は低い
- 成果物要件（「各チェックボックスに検証エビデンスがコメントとして存在すること」）: Issue コメントに検証結果を残す形式。検証の有無が可視化される
- フロー組み込み: hooks で `gh issue edit` 時にチェックは困難。スキル定義への組み込みが現実的
