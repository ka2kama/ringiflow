# レビュー返信の検討事項に Issue 追跡が欠如

## 事象

`/review-and-merge` でレビューコメント（`delete_all` の部分失敗ハンドリング）に対し、「検討します」と返信した。この返信には Issue 番号や TODO コメントなどの追跡手段がなく、検討の実施を担保する仕組みがなかった。

## 原因分析

### 直接原因

レビューコメントへの返信で「検討します」「対応を検討します」と書く場合、その検討事項を追跡する仕組みが `/review-and-merge` スキルに組み込まれていない。

### 構造的原因

`/review-and-merge` の Step 4 では、コメントへの対応を「修正する / スキップ」の 2 択で定義している。しかし実際には第 3 の選択肢「延期（今は対応しないが将来対応する）」がある。

現在のスキル定義:

| 対応 | 返信内容 | 追跡 |
|------|---------|------|
| 修正する | 「修正しました。」 | コミットで完了 |
| スキップ | 「（スキップ理由）」 | 不要 |
| 延期 | 「検討します」 | **未定義** |

「延期」は「スキップ」とは異なる。スキップは意図的な見送り（理由付き）だが、延期は将来の対応を約束している。約束を追跡する仕組みがなければ、実質的に「忘却」と同じになる。

## 対策

### 暫定対策（実施済み）

- Issue #471 を作成し、検討事項を追跡可能にした
- コードに `TODO(#471)` を追記した
- レビュー返信を Issue 参照付きに更新した

### 恒久対策

`/review-and-merge` スキルの Step 4 に「延期」対応パスを追加する。延期する場合、GitHub Issue を作成してから返信する。

返信テンプレート:

```bash
# 延期する場合（Issue を作成してから返信）
gh api "repos/{owner}/{repo}/pulls/{pr_number}/comments/{comment_id}/replies" \
  -f body="Issue #番号 で対応します。"
```

成果物要件: 「検討します」「後で対応します」等の延期返信には、必ず Issue 番号が含まれていること。
