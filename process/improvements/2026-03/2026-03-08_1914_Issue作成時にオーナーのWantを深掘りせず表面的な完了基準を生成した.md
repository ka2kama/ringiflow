# Issue 作成時にオーナーの Want を深掘りせず表面的な完了基準を生成した

## 事象

Issue #1080 の作成時、オーナーの発言「テストケースあるいはそれに準じる検証方法(型安全性など)が抜けていたこと」から、AI が以下の完了基準を生成した:

- テストリスト作成時に検証観点が抜け落ちるメカニズムの分析
- 既存のギャップ発見観点との関係整理
- 対策の設計と実装（テストリスト作成プロセスの改善、チェックリスト、観点マップ等）

その後、オーナーとの対話で3回の修正が必要になった:

1. 「まずはテストケースの抜け漏れから始める」→ スコープの絞り込み
2. 「該当なしに限った話ではない」→ 前提の修正
3. 「Want が正確に記述されてさえいれば、残りは自然と導出されるはず」→ 本質の指摘

最終的な Want は「テストリスト作成時に、本来列挙されるべきテストケースが抜け落ちないこと」に収束した。

## 原因分析

### 直接原因

オーナーの発言を深掘りせず、表面的な理解から完了基準に How（分析手順、特定の対策アプローチ）を混入させた。

### 構造的要因

Issue 作成プロセスに「オーナーの Want を検証する」ステップが存在しない。`problem-solving.md` で Want と How の区別は定義されているが、Issue 作成時にオーナーの発言から Want を抽出・検証する手順がない。

SI（システムインテグレーション）で顧客要件を深掘りするのと同様に、オーナーが発した Want が本質的なニーズを正確に表現しているとは限らない。オーナーの発言は Want の手がかりであり、そのまま完了基準に転写してよいものではない。

### 思考的要因

AI が「分析すべき問い」「既存との関係整理」など、自身の作業計画を完了基準に混入させた。これは Issue の読者（将来の実行者）ではなく、現在の自分のために書いていた。

## 対策

### Want の深掘りプロセス

Issue 作成時（特にオーナーの発言から Issue を起こす場合）に、完了基準を書く前に Want を検証する。

検証の問い:
- オーナーの発言の背後にある本質的なニーズは何か？
- 完了基準に How（特定の対策手段）が混入していないか？
- この Want を満たす別の完了基準はあり得るか？（あり得るなら、完了基準が How に偏っている兆候）

形式: 行動規範

### 転換の検討

| 対策 | 形式 | 転換可能性 |
|------|------|-----------|
| Want の深掘りプロセス | 行動規範 | 成果物要件への転換: Issue に「Want」セクションを必須化し、Want と完了基準が分離されていることを検証可能にする方法がある。ただし、Want の「正しさ」は成果物要件では検証困難（形式的に書かれていても中身が How である可能性）。フロー組み込み: Issue 作成の自動検証は困難（Want の妥当性は意味的判断が必要） |
