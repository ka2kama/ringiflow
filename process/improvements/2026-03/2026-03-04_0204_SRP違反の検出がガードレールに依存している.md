# SRP 違反の検出がガードレールに依存している

## 事象

ファイルサイズ分割 PR を振り返ったところ、全件が SRP 違反（責務の混在）を原因としていた。SRP 違反は TDD Refactor の設計原則レンズで検出すべき設計品質の問題だが、実際には 500 行チェック（`just check-file-size`）で初めて検出され、大規模な事後分割が必要になっていた。

対象 PR:

| PR | 対象 | 超過行数 | 超過の原因 |
|----|------|---------|-----------|
| #363 | `workflow.rs` | 1665行 | 3 エンティティが 1 ファイルに混在 |
| #584 | `decision.rs` / `lifecycle.rs` | 1882行 / 1289行 | 複数操作が 1 モジュールに混在 |
| #1012 | `instance.rs` | 754行 | 型定義・遷移ロジック・DB 変換の混在 |
| #1013 | `Designer.elm` | 1965行 | update/canvas/panel/toolbar の混在 |
| #1016 | `Detail.elm` | 1479行 | 承認・再提出・コメント・進捗の混在 |
| #1021 | `Update.elm` / `Canvas.elm` | 700行 / 663行 | 永続化ロジック・状態遷移の混在 |

アーキテクチャパターンの帰結やテスト肥大が原因のケースは 1 件もなく、全件が設計品質の問題だった。

## 原因分析

### 直接原因

TDD Refactor の SRP レンズ（「この関数/モジュールに変更理由は 1 つだけか？」）が、既存モジュールへの機能追加時に機能していない。

### 構造的原因

1. **変更単位と評価単位のミスマッチ**: TDD サイクルのスコープは「今書いた数十行」だが、SRP の評価はファイル全体を対象とする。各増分は小さいため、SRP レンズが「まだ大丈夫」と判断し続ける（茹でガエル問題）
2. **マクロ設計を問うタイミングの空白**: 設計原則レンズはミクロ設計（命名、重複、最小性）には有効だが、モジュール境界・責務配分といったマクロ設計の改善に結びついていない
3. **品質ゲートの検証行為の欠如**: 品質チェックリストに「責務の明確さ: 各コンポーネントの責務が単一か」の項目は存在するが、具体的な検証行為（責務の棚卸し等）に結びついていない

### 比喩的整理

500 行チェックはガードレールに相当する。ガードレールがあること自体は有用だが、ガードレールにぶつかってから方向修正する運転は理想ではない。現状は多層ゲートの中間層（L2: 実装中、L3: PR ready 前）が弱く、最終層（L4: ガードレール）に依存している。

| 層 | タイミング | 性質 | 現状 |
|----|-----------|------|------|
| L1 | 設計段階 | ステアリング（最初から正しい方向に進む） | 新規は機能。既存への追加時が弱い |
| L2 | 実装中（TDD Refactor） | 車線維持（逸脱に気づいて戻す） | ミクロ設計のみ機能。マクロは空白 |
| L3 | PR ready 前（品質ゲート） | 車線逸脱警告（俯瞰で確認） | 項目はあるが検証行為がない |
| L4 | `just check` | ガードレール（最終的な物理的制約） | 機能している |

## 対策

問題の性質: プロセス的（品質ゲートとリファクタリングフローの不備）

### 対策形式の検証

| 対策案 | 形式 | 有効性 | 懸念 |
|--------|------|--------|------|
| ~~TDD Refactor で毎回 SRP を強く意識する~~ | 行動規範 | 低 | 変更単位と評価単位のミスマッチは構造的に解消されない |
| ~~500 行閾値を 300 行に下げる~~ | フロー組み込み | 低〜中 | 対症療法。検出が早まるだけで予防にならない |
| PR ready 前の品質ゲートで責務棚卸しの検証行為を具体化する | 成果物要件 | 中〜高 | AI の強み（機械的な棚卸し）を活かせる |
| 設計段階で既存モジュールへの責務追加を評価する | 成果物要件 | 中 | 設計と実装の乖離リスク |

### 採用する対策（仮説）

多層ゲートの L3（PR ready 前の品質ゲート）を強化する方向を優先的に検討する。

具体案:
- 品質チェックリストの「責務の明確さ」に、変更したファイルの public 関数/型を列挙し責務グルーピングを確認する検証行為を追加する
- L1（設計段階）での既存モジュール影響評価も補助的に検討する

L2（TDD Refactor）の強化は、変更単位と評価単位のミスマッチという構造的問題があるため優先度を下げる。

L4（500 行チェック）はガードレールとして維持する。

注: この対策は仮説であり、実行時に精査が必要。

## 検証

PR #1027 で対策を実施。

- L3 品質チェックリストの「責務の明確さ」に、仕様突合パターンに準拠した3ステップの責務棚卸し手順を追加

- L4（500行チェック）は変更なし

対策の仮説は精査の結果、方向性は妥当と判断。対象ファイルの選定基準を「新しい public 要素を追加したファイル」に具体化した。

有効性の確認は、今後の PR で責務棚卸しが L4 より前に SRP 違反を検出するケースが出現するかで追跡する。
