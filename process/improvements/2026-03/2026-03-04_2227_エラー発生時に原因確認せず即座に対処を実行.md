# エラー発生時に原因確認せず即座に対処を実行

## 事象

PR #1021 のマージ時に `gh pr merge 1021 --squash` が `base branch policy prohibits the merge` エラーで失敗した。このエラーに対して、原因を調査せずに空コミットをプッシュして CI 再実行を試みた。

実際には CI は既に実行中であり、空コミットにより CI が最初からやり直しになった。

## 原因分析

### 直接的原因

エラーメッセージを見た直後に「CI が通っていないのが原因」と推測し、検証なしに空コミットをプッシュした。

```
正しい流れ:  エラー → 原因調査 → 検証 → 対処
実際の流れ:  エラー → 推測 → 即座に対処（空コミット）→ CI やり直し
```

### 構造的原因

troubleshooting.md には「データが先、仮説が後、修正は最後」という原則と詳細な調査フローが存在するが、これは**調査開始後**のプロセスを定義している。エラー発生直後の「調査に入る前の瞬間」にゲートがない。

散弾銃デバッグの改善記録（2026-02-20）で仮説検証ループが強化されたが、そもそも調査モードに入らずに即座に対処に飛ぶケースには対応できていない。

## 対策

### エラー時強制チェックリスト

troubleshooting.md のステップ 1（再現する）の前に、エラー発生直後の強制チェックリストを追加する。

```markdown
## ステップ 0: 立ち止まる（Stop）

エラーやコマンド失敗に遭遇したとき、対処に入る前に以下の 3 問に答える。

| # | 問い | 確認内容 |
|---|------|---------|
| 1 | エラーメッセージは何を言っているか？ | 全文を読み、キーワードを特定する |
| 2 | 現在の状態はどうなっているか？ | 関連するシステムの状態を確認する（CI の実行状態、ブランチの状態等） |
| 3 | 対処は可逆か？ | 不可逆な操作（push、削除、上書き）が含まれていないか |

3 問への回答なしに対処を実行してはならない（成果物要件: 対処の前に 3 問の回答が会話内に存在すること）。
```

この対策は成果物要件形式であり、行動規範（「確認すること」）よりも検証可能。

### 対策形式の検証

| 形式 | 該当 | 理由 |
|------|------|------|
| 行動規範 | — | 「確認すること」だけでは再発防止にならない |
| 成果物要件 | ✓ | 「3 問の回答が会話内に存在すること」で検証可能 |
| フロー組み込み | — | エラー発生は予測不能なため hooks/CI への組み込みは困難 |
