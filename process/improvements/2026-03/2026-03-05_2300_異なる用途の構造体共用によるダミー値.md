# 異なる用途の構造体共用によるダミー値

## 事象

`WorkflowInstanceDto` が一覧 API と詳細 API の両方で使用されており、一覧 API では `steps` フィールドに `Vec::new()` というダミー値が埋められていた（`backend/apps/core-service/src/handler/workflow.rs:262`）。

一覧 API のレスポンスには `"steps": []` が含まれるが、これは「ステップが0件」ではなく「ステップ情報を返していない」の意味。API 利用者はこの区別を型から読み取れない。

ユーザーからの指摘により発覚した。

## 原因分析

### 直接原因

`WorkflowInstanceDto` が一覧用と詳細用という異なる文脈で共用されており、一覧文脈では意味を持たない `steps` フィールドにダミー値を入れる必要があった。

### 構造的原因

1. 設計ブラッシュアップループのギャップ発見の観点に、「同一型の異なる文脈での再利用によるダミー値」を検出する項目がない
2. 既存の「状態依存フィールド」観点は型安全ステートマシン（ライフサイクル状態の分離）に限定されており、API レスポンス等の文脈依存の型再利用をカバーしていない
3. 品質チェックリストの「型安全性」項目も Newtype と unwrap に焦点があり、ダミー値パターンの検出に結びついていない

### 問題の構造

「状態依存フィールド」と「文脈依存のダミー値」は、いずれも "make illegal states unrepresentable" 原則の違反だが、トリガーが異なる:

| パターン | トリガー | 例 |
|---------|---------|-----|
| 状態依存フィールド | 同一エンティティのライフサイクル状態による有効フィールドの差異 | Draft 状態では `approved_by` が無意味 |
| 文脈依存のダミー値 | 同一型の異なる使用文脈による有効フィールドの差異 | 一覧 API では `steps` が無意味 |

## 対策

### 問題の性質に基づく routing

- 問題の性質: 思考的（設計レビュー時の視点の欠如）
- 対策の配置先: 設計原則（`zoom-rhythm.md` のギャップ発見の観点）

### 対策1: ギャップ発見の観点に「文脈依存のダミー値」を追加

`zoom-rhythm.md` の「欠陥の発見」セクションに、文脈依存のダミー値を検出する観点を追加する。

対策形式: 成果物要件（設計ブラッシュアップループで観点がチェックされること）

Issue #1049

### 対策2: WorkflowInstanceDto の型分離

一覧用と詳細用に DTO を分離し、ダミー値の必要性を排除する。

Issue #1050

## 次のアクション

1. `zoom-rhythm.md` のギャップ発見の観点に「文脈依存のダミー値」を追加する（Issue #1049）
2. `WorkflowInstanceDto` を一覧用と詳細用に分離する（Issue #1050）
