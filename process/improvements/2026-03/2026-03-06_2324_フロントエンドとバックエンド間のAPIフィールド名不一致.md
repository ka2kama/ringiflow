# フロントエンドとバックエンド間の API フィールド名不一致

## 事象

ドキュメント管理画面でファイルをアップロードすると、以下のエラーが発生した:

```
Failed to deserialize the JSON body into the target type: missing field `content_length` at line 1 column 151
```

フロントエンド（Elm）の JSON エンコーダが `"size"` フィールドを送信しているが、バックエンド（Rust / axum）は `"content_length"` を期待しており、デシリアライズに失敗していた。

影響範囲:
- `Api.Document.encodeUploadRequest`（ワークフロー申請のファイル添付）
- `Api.Document.requestUploadUrlForFolder`（ドキュメント管理のフォルダ内アップロード）

混入時期: PR #1031（Issue #884）で `encodeUploadRequest` を `"size"` で作成。PR #1067（Issue #885）がこのパターンをコピー。

## 原因分析

### 直接原因

Elm のエンコーダで、Elm 内部のフィールド名 `size` をそのまま JSON キーとして使用した。OpenAPI 仕様書（`openapi.yaml`）とバックエンドの Rust 構造体は `content_length` を定義していた。

### プロセス的要因（なぜ既存の開発フローが検出できなかったか）

3つのプロセスの穴が重なって検出を阻害した。

#### 1. フロントエンド Story で API テストを「該当なし」と判定

両計画（#884, #885）で全 Phase の API テスト（Hurl）が「該当なし」と判定されている。#885 の収束確認でも「フロントエンドのみのため ハンドラ/API テストは該当なし」（L232）と記載。

問題: フロントエンドが API エンドポイントを呼び出している以上、リクエストボディのフィールド名は結合レベルでしか検証できない。「フロントエンドのみ → API テスト該当なし」という判定ロジックが誤り。

#### 2. 確認事項でエンコーダとバックエンド型の突合が欠落

#884 Phase 2 の確認事項はパターンとライブラリのみで、バックエンドのリクエスト型のフィールド名を確認する項目がない。計画ファイルの As-Is（L50）には `RequestUploadUrlRequest: { filename, content_type, content_length, ... }` と正しく記載されているのに、エンコーダ実装時にこの情報が使われなかった。

#### 3. E2E テストのスコープ外判定

#885 では E2E テストが明示的にスコープ外（L80）。#884 では E2E テストリストは記載されたが、ファイルアップロードの正常系 E2E テストは実装されなかった（操作パスで E2E 層にマッピングされたにもかかわらず）。

### 構造的要因

品質チェックリストの「仕様突合」項目（`OpenAPI と実装が一致`）がバックエンド寄りの記述で、フロントエンドのエンコーダ/デコーダと OpenAPI の突合が含意されていない。また、テスト層判定の「該当なし」判定基準に、API を呼び出すフロントエンド Story での結合テストの必要性が考慮されていない。

## 対策

### 1. テスト層判定基準の明確化

現行の判定基準:

| テスト層 | 該当条件 |
|---------|---------|
| API テスト（Hurl） | 公開 API エンドポイントがある場合 |

この基準では「フロントエンド Story は API エンドポイントを作らないから該当なし」と判定される。

対策案: 判定基準に「フロントエンドが API を呼び出す場合、リクエスト/レスポンスの契約が既存の API テストで検証済みか確認する。未検証なら API テストを追加する」という観点を追加する。

形式: 成果物要件（テストリスト作成時の判定基準として `dev-flow-tdd.md` に追加）

### 2. 確認事項の観点追加

フロントエンドのエンコーダ/デコーダを書く Phase では、確認事項に「バックエンドのリクエスト型のフィールド名」を含める。

形式: 成果物要件（`pre-implementation.md` のカテゴリ2「プロジェクト内の型と構造」に、API 境界を跨ぐ型のフィールド突合を明示）

### 3. 品質チェックリストの仕様突合をフロントエンドにも拡張

現行: 「OpenAPI と実装が一致（フィールド、ステータスコード、型）」
対策案: 「フロントエンドのエンコーダ/デコーダのフィールド名が OpenAPI と一致」を明示的に追加。

形式: 成果物要件（`dev-flow-issue.md` の品質チェックリストに追加）

### 転換の検討

| 対策 | 形式 | フロー組み込みへの転換可能性 |
|------|------|--------------------------|
| テスト層判定基準 | 成果物要件 | ブラッシュアップループのギャップ発見観点に「テスト責任の断絶」が既にあるが、API テストの要否判定を強化する余地あり |
| 確認事項の観点 | 成果物要件 | 将来的に OpenAPI → Elm エンコーダの自動生成で不要になる可能性があるが、現時点では codegen が未成熟 |
| 品質チェックリスト | 成果物要件 | lefthook での自動検証は困難（JSON キー名の突合は静的解析の対象外） |
