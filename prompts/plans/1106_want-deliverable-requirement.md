### 設計判断

| # | 判断 | 選択肢 | 選定理由 | 状態 |
|---|------|--------|---------|------|
| 1 | Want フィールドの位置 | A: 概要の前 / B: 概要の後・完了基準の前 | B: Want は概要と完了基準の間に位置し、概要（何を）→ Want（なぜ）→ 完了基準（どうなったら終わり）の論理的な流れを形成する | 確定 |
| 2 | Want フィールドの required | A: 必須 / B: 任意 | A: 成果物要件の本質は「存在の保証」。任意にすると行動規範と変わらない | 確定 |
| 3 | 行動規範の扱い | A: 完全に削除して成果物要件に置換 / B: 行動規範を保持しつつ成果物要件を追加 | B: トリガー条件と検証の問いは対話全般に適用される行動指針であり、Issue 作成に限定されない。成果物要件は Issue 作成の結果を検証するもので、行動規範とは異なるレイヤー | 確定 |
| 4 | 精査での検証強化方法 | A: Issue 本文に Want セクション必須 / B: 精査コメントの Want で代替可 | A: Issue 本文自体に Want を記載することで、作成時点で分離を強制する。精査コメントは事後検証 | 確定 |

### Phase 1: Issue テンプレートに Want フィールドを追加

#### 確認事項
- パターン: 既存の Issue テンプレート構造（feature.yaml, bug.yaml, epic.yaml）→ 確認済み
- パターン: GitHub Issue Forms の textarea 構文 → 既存テンプレートに準拠

#### 変更内容

feature.yaml: 概要の後に Want フィールドを追加

```yaml
  - type: textarea
    id: want
    attributes:
      label: Want（本質的なニーズ）
      description: |
        この Issue が解決しようとしている本質的なニーズ。
        具体的な手段（How）ではなく、達成したい状態を記述してください。
      placeholder: |
        例: ユーザーが安全にログインでき、不正アクセスから保護されている状態
        NG例: POST /auth/login を実装する（これは How）
    validations:
      required: true
```

### Phase 2: ルールファイルの成果物要件への転換

#### 確認事項
- パターン: CLAUDE.md の Want 深掘りセクション → L271-305
- パターン: 既存の成果物要件パターン（problem-solving.md, troubleshooting.md, zoom-rhythm.md）→ 確認済み
- パターン: dev-flow-issue.md の精査観点 → L29-41

#### 変更内容

**CLAUDE.md（L271-305）**: 行動規範を保持しつつ、成果物要件セクションを追加

追加する成果物要件:
```markdown
### 成果物要件

Issue 作成・修正時に、Issue 本文の Want セクションが以下を満たすこと:

| # | 要件 | 検証基準 |
|---|------|---------|
| 1 | Want セクションが存在する | Issue 本文に「Want」ラベルのフィールドまたはセクションがある |
| 2 | Want が How と分離されている | Want に具体的な実装手段・技術選定・解決策が含まれていない |
| 3 | Want が完了基準と独立している | 完了基準を変えても Want は変わらない（Want は手段に依存しない） |

Issue テンプレートの Want フィールドを使用する。`gh issue create` で作成する場合も、本文に Want セクションを含める。
```

**problem-solving.md（精査テンプレート）**: 精査の観点に Issue 本文の Want セクション検証を追加

精査の観点テーブルに追加:
```markdown
| Want セクションの存在 | Issue 本文に Want セクションがあるか？ない場合は追記を提案する |
```
