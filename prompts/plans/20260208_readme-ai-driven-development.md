# README.md AI 駆動開発セクション拡充計画

## Context

既存の README.md（208行）は技術スタック・アーキテクチャ・開発フローなどが充実しているが、「AI 主導の開発」セクション（161-169行目）は 6 行と極めて簡潔。このプロジェクトの最大の特徴である「AI 駆動で品質を追求する仕組み」が第三者に伝わらない。

目的: 第三者が README を読んだだけで「AI 主導開発で品質を担保するために何をしているか」の全体像を把握できるようにする。

## 対象ファイル

- 編集: `README.md`（ルート、worktree `../ringiflow-readme` で作業）

## スコープ

### 対象

- 「AI 主導の開発」セクション（現行 161-169 行目）の拡充
- 同セクションのトップレベル昇格（`###` → `##`）

### 対象外

- 他のセクション（技術スタック、アーキテクチャ等）の変更
- Rust バッジ更新（別件として対応すべき。今回のスコープに含めない）
- Zenn 記事の執筆

## 変更方針

### 構造変更: トップレベルセクションに昇格

現状: `## 開発フロー` > `### AI 主導の開発` （サブセクション）

変更後: `## AI 駆動開発` として独立したトップレベルセクションに昇格する。

理由: このプロジェクトの最大の差別化要因であり、第三者の関心もここにある。「開発フロー」の一部では格が低すぎる。

配置: 「開発フロー」セクションの直後、「Getting Started」セクションの直前。
開発フローから AI のくだりを削除し、新セクションに移動する形。

### 新セクションの構成

```markdown
## AI 駆動開発

[概要: 役割分担]

### AI の行動規範
[CLAUDE.md + rules の説明]

### 改善フィードバックループ
[ミス → 原因分析 → ルール改訂 のサイクル]

### 品質の二重防壁
[設計段階 + レビュー段階の二段構え]
```

### 各サブセクションの内容

#### 概要（現行の役割分担を維持 + 導入文）

```markdown
## AI 駆動開発

このプロジェクトでは AI エージェント（Claude Code）に開発を主導させながら、「正しく」「高品質に」プロダクトを作るための仕組みを構築・運用している。

| 役割 | 担当 | 内容 |
|------|------|------|
| オーナー | 人間 | 方針決定、レビュー、最終判断 |
| 実装者 | Claude Code | 設計、実装、テスト、ドキュメント作成 |
| レビュアー | Claude Code Action | PR の自動レビュー |
```

#### AI の行動規範

- [CLAUDE.md](../../CLAUDE.md)（612行）と [17 のルールファイル](../../.claude/rules/) が AI の行動を構造的に規定
- 主要ルールの紹介（箇条書き 5-6 件、各1行 + リンク）:
  - Issue 駆動開発・TDD の強制
  - 実装前チェックリスト（推測で書かない）
  - 俯瞰・実装リズム（設計と実装の往復、収束確認）
  - 問題解決フレームワーク（Want → To-Be → As-Is → 根本原因）
  - ベストプラクティス起点（技術選定、方法論設計）
  - ドキュメント自動作成（ADR、ナレッジベース、セッションログ等）

#### 改善フィードバックループ

- サイクルの説明: AI のミス → 原因分析 → 改善記録 → CLAUDE.md / rules 改訂 → AI の行動変容
- 現在 34 件以上の改善記録が蓄積されていること
- Mermaid でサイクル図を描く（既存 README が Mermaid 使用済みなので統一）
- 代表的な改善事例を 3 件、タイトル + 1 行説明 + リンクで紹介:
  1. [YAGNI/KISS の拡大解釈による設計品質低下](../../process/improvements/2026-02/2026-02-01_0004_YAGNI-KISSの拡大解釈による設計品質低下.md) — AI が YAGNI を設計品質の妥協に使う問題 → 機能スコープと設計品質の区別を明文化
  2. [E2E 視点の完了基準欠如](../../process/improvements/2026-01/2026-01-29_1304_E2E視点の完了基準欠如.md) — API は動くが UI で操作できない → 完了基準に E2E 視点を追加
  3. [自己検証ループの自動実行欠如](../../process/improvements/2026-02/2026-02-05_2100_自己検証ループの自動実行欠如.md) — 行動規範ではなく成果物要件にすることで構造的に強制

#### 品質の二重防壁

- Mermaid で設計→実装→レビューのフロー図
- 最初の砦（設計段階）: ブラッシュアップループでギャップゼロまで詰める
- 最後の砦（Ready for Review）: 品質チェックリスト + Self-review セクション
- 詳細リンク: zoom-rhythm.md、Issue 駆動開発手順書
