#### 設計内容

**トリガー条件（深掘りが必要な場面）:**

| 場面 | 例 |
|------|-----|
| 問題・課題の報告を受けたとき | 「テストケースが抜けていた」→ 表面的な対処か構造的改善か |
| 方針・方向性の相談を受けたとき | 「〜を導入したい」→ 手段が先行していないか |
| 完了基準・要件を定義するとき | Issue 作成、スコープ定義 |

**深掘りが不要な場面:**

| 場面 | 理由 |
|------|------|
| 明確な実行指示 | 「この typo 直して」「テストを実行して」→ What と How が一致 |
| 承認・確認の応答 | 「y」「それでいい」→ 既に合意形成済み |
| 事実の質問 | 「このファイルはどこ？」→ 正解が一つ |

**検証の問い（トリガー該当時に実施）:**
1. この発言の背後にある本質的なニーズは何か？
2. 発言に含まれる How（特定の解決策・手段）を Want と混同していないか？
3. 提示しようとしている定義・決定は、別の形でも Want を満たせるか？（Yes なら How に偏っている兆候）

**アクション:**
- 検証の結果、Want が不明確と判断した場合: 自分の理解を提示し、ユーザーに確認する
- Want が明確な場合: そのまま進行する（毎回確認を挟まない）
