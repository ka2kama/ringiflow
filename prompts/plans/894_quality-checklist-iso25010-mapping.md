# 計画: #894 品質チェックリスト全体と ISO 25010 品質特性の対応見直し

## Context

#891 で設計原則レンズの検証項目を見直した際、品質チェックリスト全体と ISO 25010 品質特性の体系的なマッピングが未実施であることが判明した。品質チェックリスト（6.2）の各項目がどの品質特性に対応するかを明示し、カバレッジギャップを特定して対応方針を決定する。

## 設計判断

### 1. マッピング表の配置場所

採用: `docs/06_ナレッジベース/methodology/ISO25010.md` にマッピングセクションを追加し、品質チェックリストから相互参照リンクを設置する。

不採用: チェックリスト本体にインラインでタグ付け。理由: チェックリスト本体は実用チェックリストとして軽量性を重視する。インラインの品質特性タグは可読性を低下させる。

### 2. 低優先品質特性のギャップ対応

採用: 意図的除外として記録のみ（性能効率性・互換性・柔軟性・安全性）。理由: ISO25010.md の優先度表で「低優先: 現時点では要件なし」と判断済み。使わない項目をチェックリストに追加すると形骸化リスクが上がる。

### 3. チェックリストのカテゴリ再編成

採用: 現在のプロセス指向カテゴリ（A-H）を維持する。理由: ISO 25010 に沿った再編成をすると、同じ品質特性の項目がドキュメント・テスト・コード品質にまたがり、確認フローが壊れる。

### 4. セキュリティ項目の追加

セキュリティは「現在の重点」にもかかわらず品質チェックリストに明示的な項目がゼロ。「コード品質確認（マイナス→ゼロ）」に基本的なセキュリティ確認項目を 1 つ追加する。

### 5. マッピングの粒度

採用: 副特性レベルでマッピングし、サマリーは品質特性レベル。理由: 品質特性レベルだけでは同じ「保守性」内の違いが見えず、カバレッジギャップの分析精度が不十分。

#### サマリー表

| ISO 25010 品質特性 | チェックリストカテゴリ | カバー状況 | 備考 |
|-------------------|---------------------|-----------|------|
| 保守性 | 設計・ドキュメント、テスト、コード品質、品質向上 | 強 | 最多項目。解析性・試験性・修正性・モジュール性を幅広くカバー |
| 機能適合性 | Issue との整合、テスト、コード品質、横断検証 | 強 | 完全性・正確性を中心にカバー |
| 操作性 | UI/UX 品質、デザイン品質向上 | 強（FE 変更時） | フロントエンド変更がある場合のみ適用 |
| 信頼性 | コード品質、品質向上、UI/UX 品質 | 部分的 | 型安全性・状態表示経由。可用性・回復性は未カバー |
| セキュリティ | コード品質 | 基本的 | 本 Issue でセキュリティ基本確認を追加 |
| 性能効率性 | — | 意図的除外 | Phase 3 以降で追加予定 |
| 互換性 | — | 意図的除外 | 現時点で要件なし |
| 柔軟性 | 品質向上 | 微弱 | ベストプラクティス項目が間接的にカバー |
| 安全性 | — | 意図的除外 | SaaS 特性上、物理安全性は対象外 |

#### カバレッジギャップ分析

| 品質特性 | カバーされていない副特性 | 対応方針 |
|---------|----------------------|---------|
| 信頼性 | 可用性、回復性 | Phase 2 以降でインフラ・運用の信頼性項目を追加予定 |
| セキュリティ | 否認防止、責任追跡性、耐性 | 高度なセキュリティ項目は専用チェックリストを Phase 2 以降で検討 |
| 性能効率性 | 全副特性 | 意図的除外。Phase 3 以降で追加予定 |
| 互換性 | 全副特性 | 意図的除外。現時点で要件なし |
| 柔軟性 | 拡張性、設置性、置換性 | 意図的除外。IaC（Terraform）で設置性は担保されるが、チェックリスト項目としては不要 |
| 安全性 | 全副特性 | 意図的除外。SaaS 特性上、物理安全性は対象外 |
