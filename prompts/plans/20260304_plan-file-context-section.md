# 計画ファイルにコンテキストセクションを必須化する

## 背景

計画ファイルの必須要素（zoom-rhythm.md）は検証・品質保証に重点が置かれている（確認事項、テストリスト、操作パス、ブラッシュアップループ、収束確認）。一方、新セッションで計画ファイルを読んだときに「なぜこれをやるのか」「どこまで進んだか」「どのブランチで作業しているか」が分からず、自律的に作業を再開できない。

既存の改善記録でも、コンテキストクリア後に計画ファイルのコミット漏れ（2026-02-16）や Draft PR 作成漏れ（2026-02-24）が構造的に発生している。

## 対策

計画ファイルの冒頭に「コンテキスト」セクションを必須化し、新セッションが計画ファイル 1 つで作業を再開できるようにする。

## 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `.claude/rules/zoom-rhythm.md` | `## 計画ファイルへの記載（必須）` に `### コンテキスト（ファイル冒頭）` を追加 |
| `prompts/plans/README.md` | 計画ファイルの構造セクションを追加 |

## 変更内容

### 1. zoom-rhythm.md（L178 の後、L180 `### 確認事項` の前に挿入）

以下のサブセクションを追加する:

```markdown
### コンテキスト（ファイル冒頭）

新セッションが計画ファイル 1 つで作業を再開できるよう、以下をファイル冒頭（スコープや Phase 定義より前）に記載する。

` ` `markdown
## コンテキスト

### Issue
- 番号: #NNN
- Want: [本質的なニーズ（1〜2 文）]
- 完了基準:
  - [完了基準1]
  - [完了基準2]

### ブランチ / PR
- ブランチ: `feature/NNN-xxx`
- PR: #NNN（Draft）

### As-Is（探索結果の要約）
[新セッションが再探索なしに作業を続けるために必要な情報: キーとなるファイルパス、発見したパターン、既存コードの構造]

### 進捗
- [ ] Phase 1: [名前]
- [ ] Phase 2: [名前]
` ` `

各項目の記載基準:

| 項目 | 記載すること | 記載しないこと |
|------|------------|--------------|
| Want | Issue が解決する本質的なニーズ | 実装手段（How） |
| 完了基準 | Issue 本文の完了基準をインラインでコピー | Issue 番号への参照のみ |
| ブランチ/PR | ブランチ名と PR 番号。未作成なら「未作成」 | — |
| As-Is | 重要なファイルパス・パターン・制約の要約 | Phase 確認事項と重複する詳細 |
| 進捗 | Phase の完了状態。完了したら `[x]` に更新 | TDD サイクル内のステップ詳細 |

**禁止:**
- 完了基準を Issue 番号への参照のみで済ませること
- コンテキストセクションのない計画ファイルで実装を開始すること
```

注: ` ` ` は実際には連続するバッククォート 3 つ（Markdown のネスト制約のためスペース区切りで表記）。

### 2. prompts/plans/README.md（末尾に追加）

```markdown
## 計画ファイルの構造

→ 必須要素の詳細: [zoom-rhythm.md > 計画ファイルへの記載](../../.claude/rules/zoom-rhythm.md#計画ファイルへの記載必須)

| セクション | 配置 | 役割 |
|-----------|------|------|
| `## コンテキスト` | ファイル冒頭 | Issue・ブランチ・As-Is・進捗（セッション再開用） |
| スコープ・設計判断 | コンテキストの後 | 対象/対象外、設計の選択肢と理由 |
| Phase ごとの確認事項 | Phase 内 | 型・パターン・ライブラリのゲート条件 |
| Phase ごとの操作パス | Phase 内 | テスト設計のトップダウン入力 |
| Phase ごとのテストリスト | Phase 内 | テストピラミッド全層 |
| ブラッシュアップループの記録 | ファイル末尾 | 設計の試行錯誤の記録 |
| 収束確認（設計・計画） | ファイル末尾 | 設計の完成度チェック |

### コンテキストの更新タイミング

| タイミング | 更新項目 |
|-----------|---------|
| ブランチ・PR 作成時 | ブランチ/PR 情報 |
| Phase 完了時 | 進捗チェックボックスを `[x]` に |
| 新たな重要発見時 | As-Is に追記 |
```

## 設計判断

### 進捗追跡の方式: コンテキスト内チェックリスト

Phase 見出しにステータスを埋め込む案（`## Phase 1 [完了]`）は、見出しリンクが壊れるため却下。コンテキストセクション内にチェックリスト形式で配置する。

### As-Is の範囲: Phase 確認事項との分担

- コンテキストの As-Is: 探索で得た「重要な発見の要約」（ファイルパス・パターンレベル）
- Phase の確認事項: 実装前の「ゲート条件としての詳細」（型シグネチャ・コードスニペット）

重複を最小化しつつ、新セッションが「どのファイルを開くべきか」を知れるようにする。

### 完了基準: インラインコピー vs 参照

Issue 番号への参照のみでは、新セッションが Issue を別途読みに行く必要がある。完了基準はインラインでコピーし、Issue 側が更新された場合は計画ファイルも更新する。

## 検証方法

1. zoom-rhythm.md の変更後、`## 計画ファイルへの記載（必須）` セクションの構造が整合していることを目視確認
2. README.md の構造表が zoom-rhythm.md の必須要素と一致していることを確認
3. 既存の計画ファイル（例: #1004）と新テンプレートを比較し、追加される情報が明確であることを確認
