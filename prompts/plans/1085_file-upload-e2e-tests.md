### As-Is（探索結果の要約）

アップロードインフラ:
- MinIO が Docker で稼働（dev: port 19000, API test/E2E: port 19002）
- Presigned URL 方式: BFF が URL 発行 → ブラウザが S3 に直接 PUT → BFF に完了通知
- E2E テスト環境（`run-e2e.sh`）は MinIO を起動するため、実際のアップロードが可能

ドキュメント管理画面（`/documents`）:
- フォルダツリー + ファイル一覧レイアウト
- フォルダ作成（管理者のみ）→ フォルダ選択 → 「アップロード」ボタン → ファイル選択ダイアログ
- `File.Select.file` で `<input type="file">` を動的生成 → Playwright の `filechooser` イベントで対応
- 成功時「ファイルをアップロードしました」メッセージ表示
- シードデータにフォルダなし → テスト内でフォルダ作成が必要

ワークフロー申請（`/workflows/new`）:
- 動的フォーム内の `file` タイプフィールドで `Component.FileUpload` を使用
- ドラッグ&ドロップ / クリック選択（`File.Select.files`）
- フロー: ファイル選択 → Pending → 下書き保存 → workflowInstanceId 取得 → アップロード開始 → Completed → 申請可能
- シードデータの定義（汎用申請、2段階承認）にファイルフィールドなし
- テストにはファイルフィールドを含む定義が必要 → シードデータ追加が最もクリーン

既存 E2E テストパターン:
- 認証: `auth.setup.ts` で storageState 保存、`ADMIN_AUTH_FILE` で復元
- テストデータ: `helpers/test-data.ts` にシード定数
- ロケーター: `getByRole` > `getByText` > `getByLabel` > `locator` (CSS)
- ユニーク化: `Date.now()` でテスト間衝突を防止

キーファイル:
- `frontend/src/Component/FileUpload.elm`: 共有アップロードコンポーネント
- `frontend/src/Page/Document/List.elm`: ドキュメント管理画面
- `frontend/src/Page/Workflow/New/Update.elm`: ワークフロー申請のアップロード処理
- `frontend/src/Api/Document.elm`: API クライアント
- `tests/e2e/helpers/workflow.ts`: ワークフロー操作ヘルパー

## 仕様整理

### スコープ
- 対象:
  - ドキュメント管理画面でのファイルアップロード（フォルダ作成 → ファイル選択 → アップロード → 一覧に表示）
  - ワークフロー申請でのファイル添付（ファイル選択 → 下書き保存 → アップロード完了 → 申請 → 詳細画面で確認）
  - E2E テスト突合表の更新
- 対象外:
  - ファイルダウンロード（クリティカルパスの範囲外、別途テスト追加可）
  - ファイル削除（クリティカルパスの範囲外）
  - バリデーションエラー（E2E ではクリティカルパスのみ。バリデーションはユニットテスト責務）
  - ドラッグ&ドロップ（Playwright でのシミュレーションが不安定。クリック選択で同じコードパスをカバー）
