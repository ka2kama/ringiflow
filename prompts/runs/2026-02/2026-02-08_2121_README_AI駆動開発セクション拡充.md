# README AI 駆動開発セクション拡充

## 概要

README.md の「AI 主導の開発」セクション（6行）を「AI 駆動開発」として独立したトップレベルセクションに昇格・拡充した。第三者が README を読んだだけで、AI 駆動開発で品質を担保するために何をしているかの全体像を把握できるようにした。

## 背景と目的

このプロジェクトの最大の特徴は AI エージェント主導で品質を追求する仕組みにあるが、README ではわずか 6 行のサブセクションで触れるのみだった。AI に興味がある第三者エンジニアが README を訪れた際に、その仕組みの全体像が伝わらない状態だった。

## 実施内容

### README.md の構造変更

1. `### AI 主導の開発`（開発フローのサブセクション、6行）→ `## AI 駆動開発`（独立トップレベルセクション、約60行）に昇格・拡充
2. セクション位置をプロジェクト理念の直後（技術スタックの前）に移動。AI に興味がある読者が離脱する前に核心に到達できるようにした
3. Problem statement を冒頭に追加:「AI にコードを書かせること自体は容易になった。しかし…」

### その他の整理

## 設計上の判断

| 判断 | 選択 | 理由 |
|------|------|------|
| セクション位置 | プロジェクト理念の直後 | プロジェクトの差別化要因を技術詳細より先に提示。AI に興味がある読者が離脱する前に核心に到達させる |
| Problem statement | 冒頭 2 行で課題感を提示 | HOW だけでなく WHY がないと「なぜこの仕組みが必要か」が伝わらない |
| Outcome セクション | 追加しない | 定量データがないため空疎になるリスク。改善事例 3 件が暗黙に成果を示している |
| 品質の記述 | 「2 箇所で担保」→「全体を通じて作り込む」 | TDD を含む実装中の品質維持が抜け落ちていた |
| 改善事例数 | 3 件 | 多すぎず少なすぎず、異なる軸（設計品質 / 完了基準 / プロセス改善）のバリエーション |
| Mermaid 使用 | 使用する | 既存 README がアーキテクチャ図で使用済み。統一 |

## 議論の経緯

### セクション位置の昇格

初版では `## 開発フロー` の直後に配置していたが、ユーザーから「AI に興味がある第三者に十分伝わるか」という観点で再評価を求められた。第三者の読み体験を分析した結果、AI セクションが 160 行目にあるのは遠すぎると判断し、プロジェクト理念の直後（技術スタックの前）に移動した。

### Problem statement の必要性

AI セクションが HOW（仕組みの説明）に終始しており、WHY（なぜ必要か）がないことを指摘。「AI にコードを書かせること自体は容易になった。しかし…」という課題感を 2 行で追加した。Outcome（成果）セクションは定量データがないため追加しないと判断。

### 品質の記述精度

「品質は設計段階と実装後の 2 箇所で担保する」に対して、ユーザーから「そこだけでもない」という指摘。実装中の TDD も品質担保に寄与しているため、「品質は開発フロー全体を通じて作り込む」に修正。

### 計画ファイルの配置

計画ファイルがメインの worktree に残っており、`docs/readme` ブランチの worktree に含まれていないことをユーザーが発見。worktree は作業ディレクトリが独立しているため、手動でコピーしてコミットした。

### 独自用語の一般化

README にプロジェクト独自の用語（Want、ブラッシュアップループ、俯瞰・実装リズム、砦）がそのまま残っていた。第三者には意味が伝わらないため、一般的な用語に置換した。

### トーンの修正

ユーザーから「上から目線感がある」との指摘。学習&実験プロジェクトにもかかわらず、README の文体が「解決済み」の権威的なトーンになっていた。謙虚で事実を述べる文体に修正した。

### 語りかけ文の排除

README は技術記事ではなく、事実を簡潔に伝える場。語りかけ文（Problem statement 含む）や修辞的な文を削除し、事実ベースの記述に統一した。

### テキストの簡潔化

全体を通して冗長な表現を簡潔にした。導入文の削除（セクション見出しと内容で自明な場合）、太字の除去、重複表現の統合を実施。

## 学んだこと

- README のセクション順序は「最も伝えたいことを最初に」が原則。技術スタック → アーキテクチャという定型構造に囚われず、プロジェクトの差別化要因を先に持ってくるべき
- README は技術記事ではない。語りかけ文や修辞的な文は不要。事実を簡潔に述べる
- プロジェクト独自の用語は、内部ドキュメントでは有用でも README では障壁になる。第三者の読者を意識して一般的な用語を使う
- 学習・実験プロジェクトのトーンは「解決した」ではなく「取り組んでいる」が適切
- git worktree で作業する際、plan mode のファイルはメインの worktree に生成される。作業ブランチの worktree には手動でコピーが必要
