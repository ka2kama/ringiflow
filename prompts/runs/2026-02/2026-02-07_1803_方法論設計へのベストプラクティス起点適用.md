# 方法論設計へのベストプラクティス起点適用

## 概要

Issue #271 に基づき、「ベストプラクティス起点」の原則を方法論・プロセス設計にも明示的に適用した。ルールファイル、ナレッジベース、既存フレームワークへの注釈を作成し、独自フレームワークと既知手法の対応関係を文書化した。

## 背景と目的

CLAUDE.md の「ベストプラクティス起点」は「全領域に適用...例外なし」と記載されていたが、具体化ドキュメントは技術・ツール領域（`latest-practices.md`）のみだった。problem-solving.md や zoom-rhythm.md は複数の既知手法を実践から帰納的に再発明したものだったが、既知手法との照合が行われていなかった。

前回セッション（`2026-02-06_2300_方法論とベストプラクティス起点.md`）で対応関係の分析と Issue 作成が完了しており、本セッションで実装を行った。

## 設計上の判断

### 「地図」メタファへの方針転換

初版の `methodology-design.md` は「まず既知のフレームワークを調査してから設計する」「帰納的に再発明するのではなく、演繹的に適用する」と記載していた。ユーザーから「既知手法にこだわりすぎていたら今の ringiflow にはたどり着けなかった」という指摘を受け、方針を修正した。

修正後の方針: 既知手法は「地図」として活用する。地図は現在地と方角を知るためのもので、地図通りに歩くことが目的ではない。演繹（既知手法 → 適用）と帰納（実践 → パターン発見）の両方を尊重する。

判断理由:
- ringiflow の独自フレームワークは、複数の既知手法を横断的に統合し、プロジェクト固有の痛みから独自要素を加えた結果。演繹一辺倒ではこの形に到達しなかった
- 技術領域と方法論領域では「ベストプラクティス起点」の意味合いが異なる。ライブラリの API は最新版に従うべきだが、方法論は「知って参照する」と「従う」の間にグラデーションがある

### ADR 不要の判断

既存原則（「全領域に適用...例外なし」）の明示化であり、新規の技術選定・設計判断ではないため、ADR は作成しなかった。

## 議論の経緯

### 「地図」メタファへの転換

ユーザーから「既知手法にこだわりすぎていたら今の ringiflow にはたどり着けなかったのでは」という問いが出た。分析の結果、ringiflow の独自性（Want の位置づけ、二次元ギャップ、複数手法の横断統合）は帰納的プロセスから生まれたものであり、演繹一辺倒では到達しなかった可能性が高いと結論した。

これを受けて `methodology-design.md` の方針を修正:
- 「既知手法を起点とする」→「既知手法を地図として活用する」
- 「帰納的に再発明するのではなく、演繹的に適用する」→ 削除し、演繹と帰納の両方を尊重する記述に変更

## 学んだこと

- 「ベストプラクティス起点」を方法論に適用する際は、技術領域とは異なるニュアンスが必要。技術は「最新版に従う」が基本だが、方法論は「知って参照する」が基本
- 帰納的プロセス（実践 → パターン発見 → 形式知化）と演繹的プロセス（既知手法 → 適用 → 調整）は相補的。どちらか一方に偏ると、現場適合度または理論的裏付けのいずれかを失う
- 独自フレームワークの知的ルーツを知ることの主な価値は「従うため」ではなく「なぜ効くのかを理解するため」と「関連文献へのアクセスを得るため」
