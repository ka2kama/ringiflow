# デザインガイドライン拡充

## 概要

Issue #652 として、`frontend/src/styles.css` のデザインガイドラインを拡充した。#644（UI/UX 品質プロセス導入）で追加されたレンズが参照する「正解の定義」として、スペーシング・レイアウト・色使い・レスポンシブの判断基準を形式知化した。

## 実施内容

### 背景分析

- #644 で守り（Nielsen's Heuristics, WCAG）の品質レンズが追加されたが、レンズが参照する「デザインの正解」が不足していた
- オーナーがデザイン領域の専門外であるため、AI が一貫した判断を行うには形式知が必要
- 守り（欠陥検出）と攻め（設計改善）の品質を追求する方針を確認

### Issue 体系

- #652: デザインガイドライン拡充（本セッション — 基盤）
- #653: デザイン品質の攻めレンズ導入（#652 に依存 — プロセス）
- Epic #445 のタスクリストを更新

### styles.css ガイドライン追加

既存8ページ + 10コンポーネントを探索し、暗黙のパターンを発見・形式知化した。新しいルールの発明ではなく、既に一貫して使われているパターンの記録。

追加した5セクション（CSS コメント形式、約120行）:

1. デザイン原則（4原則: 透明性、一貫性、意図的な余白、最小の強調）
2. スペーシング（4px グリッド準拠、コンテキスト別の値）
3. レイアウト構成（ページ骨格、コンテンツ構造、グリッドパターン）
4. 色の使い分け（6色パレットのセマンティック用途）
5. レスポンシブ（モバイルファースト、sm/lg ブレークポイント戦略）

### 理論的背景

CRAP 原則（Contrast, Repetition, Alignment, Proximity）を参照フレームワークとしたが、ガイドライン記述自体は実践的・具体的にした。

## 判断ログ

- デザイン原則の「最小のコントラスト」を「最小の強調」に変更した（CRAP の Contrast と紛らわしいため）
- CORE-06 のブレークポイント 768px と Tailwind sm(640px) の差異を確認し、Tailwind のデフォルトを採用（md(768px) は利用可能だが現在未使用）
- frontend-design Claude Code プラグインを評価・不採用とした（Creative/Consumer 向けで Enterprise SaaS に不適合）
