# 品質ゲート用語統一

## 概要

「Ready for Review」が PR 操作（`gh pr ready`）と品質チェックプロセスの両方を指していた用語の曖昧さを解消した。品質チェックポイント（品質チェックリスト + Self-review + 収束確認）を「品質ゲート（Quality Gate）」と命名し、PR 操作と明確に区別した。

## 背景と目的

Issue #350。`zoom-rhythm.md` の「品質を守る二つの砦」で「最後の砦」を「Ready for Review」と呼んでいたが、Ready for Review は `gh pr ready` で Draft を解除する操作に過ぎない。品質を担保しているのは収束確認・品質チェックリスト・Self-review であり、これらを包括する専用の名称が必要だった。

## 実施内容

1. Grep で全 md ファイルの「Ready for Review」使用箇所を網羅的に検索
2. 品質プロセスの意味で使われている箇所と PR 操作の意味で使われている箇所を分類
3. 用語候補を検討し、ユーザーと「品質ゲート（Quality Gate）」を選定
4. 5ファイルを横断的に更新

## 設計上の判断

### 用語の選定: 品質ゲート（Quality Gate）

候補: 品質ゲート、最終検証、提出前レビュー、出荷判定

「品質ゲート」を選択した理由:
- SonarQube 等で確立されたソフトウェア品質管理の用語
- ISO/IEC 25010 を参照するこのプロジェクトとの親和性が高い
- 「通過すべき関門」の意味が明確で、PR 操作と混同しにくい

### 変更対象の判断

- `prompts/` 配下（セッションログ、改善記録、計画ファイル）は歴史的記録として変更しない
- `docs/70_ADR/013_Draft_PR運用の導入.md` は「Ready for Review」を PR 操作として正しく使用しており変更不要

## 議論の経緯

### 用語の選定

4つの候補（品質ゲート、最終検証、提出前レビュー、出荷判定）を提示し、ユーザーが「品質ゲート（推奨）」を選択。SonarQube 等での確立度と、プロジェクトの ISO/IEC 25010 準拠方針との親和性が決め手となった。

## 学んだこと

- 同じ用語が操作とプロセスの両方を指す曖昧さは、ドキュメントが増えるほど混乱が拡大する。早期の用語整理が重要
- 品質プロセスに専用の名称を付けることで、プロセスの独立性と重要性が明確になる
