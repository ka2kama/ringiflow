# 方法論とベストプラクティス起点

## 概要

独自に構築した問題解決フレームワークや開発プロセスを、既知のフレームワークと比較・照合し、「ベストプラクティス起点」の原則を方法論にも適用すべきという方針を確認した。

## 背景と目的

CLAUDE.md の「ベストプラクティス起点」は技術選定やコード設計に適用していたが、方法論・プロセス設計には明示的に適用していなかった。独自フレームワーク（problem-solving.md、zoom-rhythm.md）が車輪の再発明になっていないか検証したいという動機から議論が始まった。

## 実施内容

### 独自フレームワークと既知手法の対応関係の整理

| 独自フレームワーク | 対応する既知手法 |
|---|---|
| problem-solving.md (Want → To-Be → As-Is → Gap → Root Cause → Solution) | ギャップ分析、A3 Thinking（トヨタ）、なぜなぜ分析（5 Whys） |
| zoom-rhythm.md（理想駆動 + 往復リズム） | OODA ループ、ダブルループ学習（Argyris）、リフレクティブ・プラクティス（Schön） |
| Issue 駆動 + Phase 分割 + TDD | Spec-Driven Development（GitHub Spec Kit）と多くの要素が重複 |

### SDD（Spec-Driven Development）の調査

GitHub が 2025 年後半に OSS 公開した Spec Kit を調査。パイプライン（Intent → Spec → Plan → Tasks → Implement → PR）と ringiflow の開発プロセスを比較し、要件定義書・plan mode・Issue 駆動・TDD など多くの要素がすでに存在することを確認した。

### 「Want」の既知概念との対応

problem-solving.md で独自に「Want」と呼んでいた概念の既知の対応物を調査:

- Interest（Fisher & Ury『Getting to Yes』）— Position（立場）の奥にある本当の関心。最も近い
- Job to be Done（Christensen）— 顧客が本当に達成したいこと
- Desired Outcome（Outcome-Driven Innovation）— 成果物ではなく達成したい成果

## 議論の経緯

### 独自フレームワークの妥当性

既知のフレームワーク（空・雨・傘、ロジックツリー等）を取り入れるべきかという問いから議論が始まった。比較の結果、独自フレームワークは複数の確立された手法の要素を実践から帰納的に再発明したものであることが分かった。足りないのはフレームワーク自体ではなく、分析の道具（MECE、5 Whys）であるという結論に至った。

### ベストプラクティス起点の適用範囲

CLAUDE.md の「ベストプラクティス起点」が技術選定にのみ適用され、方法論には適用されていなかったことに気づいた。新しいプロセスを作る際はまず既存フレームワークを調査してから設計する、という方針を idea Issue として記録した。

### Want の概念の位置づけ

「Want」という独自用語が交渉学の「Interest」（Fisher & Ury）に最も近いことを確認。独自用語を使うこと自体は問題ないが、既知の名前を知っていることで先人の知見にアクセス可能になるという点が重要。

### 会話コンテキストのドキュメント混入

Issue 作成時に会話中の主観的評価（「筋は良い」等）がそのまま混入し、指摘を受けて修正した。会話モードとドキュメント作成モードの切り替えが不十分であった。

## 学んだこと

- 実践から帰納的に導いたフレームワークは、演繹的に学んだものより現場適合度が高い場合がある。ただし知的ルーツを知ることで「なぜ効くのか」の理解が深まり、検索可能性も向上する
- 「ベストプラクティス起点」はコード・技術だけでなく、方法論・プロセス設計にも適用すべき普遍的な原則
- 会話での応答とドキュメントの記述は読者と目的が異なる。モード切り替えを意識的に行う必要がある
