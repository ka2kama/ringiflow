# #1091 ApiResponse 自動アンラップ実装

## 概要

- Issue: #1091
- ブランチ: `feature/1091-auto-unwrap-api-response`
- PR: #1092

BFF の `ApiResponse<T>` による `{"data": ...}` ラッピングに対応する `Decode.field "data"` を `Api.elm` の `expectJson` に集約し、呼び出し側から手動付与を排除した。

## 実施内容

### Phase 1: Api.elm のリファクタリング

- `expectJson` に `Decode.field "data" decoder` を組み込み、自動アンラップを実現
- `PaginatedResponse` 用に `expectJsonRaw`（auto-unwrap なし）を追加
- `getRaw` 関数を公開し、`ApiResponse` 以外のレスポンス形式に対応

### Phase 2: Data モジュール・Auth デコーダーの修正

- 全 Data モジュール（10 ファイル）のデコーダーから `Decode.field "data"` を除去
- `Api/Auth.elm` の `csrfTokenDecoder` と `userDecoder` から `"data"` アクセスを除去
- `Api/AuditLog.elm` を `Api.get` → `Api.getRaw` に変更

### Phase 3: テストの修正

- 全デコーダーテスト（11 ファイル）の JSON から `"data"` ラッパーを除去
- `"data" フィールドがない場合はエラー` テストケースを削除（`"data"` の責務は `Api.elm` に移動したため）

## 設計判断

| 判断 | 選択 | 理由 |
|------|------|------|
| `PaginatedResponse` の扱い | `getRaw` 関数を追加 | API 契約を変更せず、影響が Elm 側に閉じる |
| `detailDecoder`/`listDecoder` の名前 | 維持（定義を簡素化） | 呼び出し側の変更が不要 |
