# PR レビュープロンプトのベストプラクティス起点再設計

## 概要

PR レビュープロンプト（`claude-auto-review.yaml`、`claude-rules-check.yaml`）が `methodology-design.md` の「既知手法を地図として把握した上で設計する」原則に基づいていなかった問題を解決した。直近 20 PR の claude[bot] レビューコメントを実測分析し、既知のコードレビューベストプラクティス 5 つを調査した上で、重大度モデルの再設計と Verification 観点の再構成を実施した。

Issue: #1036 / PR: #1037

## 実施内容

### 実測データの収集・分析

直近 20 PR の claude[bot] コメントを分析し、以下の問題を定量的に特定した:

- Rules Check: 全違反が Critical 扱い → ドキュメントフォーマット違反で action-required: true
- Verification: Critical 0、High 0、Medium 2（1 PR のみ）。セキュリティ・パフォーマンスの検出実績ゼロ
- LGTM コメントの大量残存（再レビュー時に全 4 種類再実行）

### 既知手法の調査

以下の 5 手法を調査し、ナレッジベースに対応関係を記録した:

| 手法 | ringiflow への適用 |
|------|-------------------|
| Google Code Review Guidelines | Verification の 7 観点の根拠 |
| Conventional Comments | blocking/non-blocking 2 階層モデル |
| Bacchelli & Bird (2013) | コード改善を non-blocking 固定（実際の最多成果） |
| SmartBear Cisco Study | コメント数ガイドライン（alert fatigue 防止） |
| AI コードレビュー研究 | 確信度閾値、AI 限界の明示 |

### Rules Check 重大度モデル再設計

全違反 Critical → 2 階層に変更:

| 違反の種類 | 重大度 | blocking |
|-----------|--------|---------|
| 禁止事項違反（`**禁止:**`） | High | blocking |
| その他のルール違反 | Medium | non-blocking |

`determine-rules-check-status.sh` が result marker（`<!-- rules-check-result:pass/fail -->`）を使用しており、severity metadata ではないことを確認。プロンプト層のみの変更で完結。

### Verification 観点再構成

6 観点 → 7 観点に再構成。blocking（1-5）と non-blocking（6-7）に分離:

1. 正確性・ロジック（Google: Functionality）
2. セキュリティ（高確信度のみ、AI 研究: FPR 最小化）
3. 型安全性・堅牢性（プロジェクト固有）
4. テストの妥当性（Google: Tests）
5. 設計・複雑さ（Google: Design + Complexity）
6. パフォーマンス（non-blocking 固定、SmartBear: 確信度閾値）
7. コード改善（non-blocking 固定、Bacchelli: 実際の最多成果）

確信度閾値、コメント数ガイドライン、AI レビューの限界セクションを追加。

### 承認基準の更新

CLAUDE.md と dev-flow-issue.md に Rules Check の承認基準を追記。

## 判断ログ

- 設計判断: パフォーマンスを blocking → non-blocking に変更（理由: 実測で検出実績ゼロ、SmartBear の確信度閾値の考え方に基づき、明白なボトルネックのみ指摘）
- 設計判断: コード改善を non-blocking 固定に設定（理由: Bacchelli & Bird の研究で実際のコードレビュー成果の最多カテゴリ。価値はあるがブロッキングにすべきでない）
- 設計判断: Conventional Comments フォーマットの全面採用を見送り（理由: 既存のメタデータ方式との二重管理になる。blocking/non-blocking の概念のみ採用）

## 議論の経緯

ユーザーが PR レビュープロンプトのベストプラクティス起点化の欠如を指摘。3 つのアプローチ（A: ナレッジベース記録、B: プロンプト再設計、C: 過去レビュー分析）を提案し、ユーザーは全 3 つの実施を指示した。スコーピングの判断はすべて委任された。

実測データの分析結果（重大度インフレ、低検出率、コメント膨張）が再設計の方向性を裏付け、5 つの既知手法の調査結果と合わせて ADR-063 に設計判断を記録した。
