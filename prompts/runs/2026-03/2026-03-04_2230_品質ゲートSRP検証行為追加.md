# 品質ゲートの「責務の明確さ」に具体的な検証行為を追加

Issue: #1026
PR: #1027
日付: 2026-03-04

## 概要

品質チェックリストの「責務の明確さ」項目に、AI エージェントが機械的に実行可能な責務棚卸し手順を追加した。合わせて、設計段階のギャップ発見の観点に「責務の蓄積」を追加した。

## 背景

ファイルサイズ分割 PR 6件（#363, #584, #1012, #1013, #1016, #1021）がすべて SRP 違反を原因としており、いずれも L4（500行ガードレール）で初めて検出されていた。品質チェックリストの「責務の明確さ: 各コンポーネントの責務が単一か」には検証行為がなく形骸化していた。

→ 改善記録: `process/improvements/2026-03/2026-03-04_0204_SRP違反の検出がガードレールに依存している.md`

## 変更内容

### L3: 品質チェックリスト（`dev-flow-issue.md`）

「責務の明確さ」項目に仕様突合のサブステップパターンに準拠した3ステップの検証手順を追加:

1. 新しい public 要素を追加したプロダクションコードファイルを特定
2. 対象ファイルの全 public 要素を「変更理由」でグルーピング（レイヤー別の手がかり付き）
3. グループが2つ以上 → 分離を検討し、判断を PR 本文に記載

### L1: ギャップ発見の観点（`zoom-rhythm.md`）

欠陥の発見テーブルに「責務の蓄積: 既存モジュールへの機能追加で、既存の責務と異なる関心事が導入されるもの」を追加。

### ISO 25010 マッピング（`docs/80_ナレッジベース/methodology/ISO25010.md`）

「責務の蓄積」を保守性 > モジュール性にマッピング。

## 判断ログ

### 対象ファイルの選定基準

全変更ファイルではなく「新しい public 要素を追加したファイル」に限定。過去6件のファイル分割 PR はすべて「既存モジュールへの関数/型の追加」が原因であり、このパターンを直接捕捉する基準が最も効果的と判断。

### L2（TDD Refactor）の非強化

TDD サイクルのスコープ（数十行）と SRP 評価のスコープ（ファイル全体）のミスマッチは構造的問題であり、L2 の強化では根本解決しない。L3（PR ready 前の俯瞰）が適切な介入点。
