# セッションログ: 大型ファイル分割計画の策定

- 日時: 2026-03-02
- Issue: #996

## 概要

健全性診断で検出された大型ファイル（1000行超）6件について、責務分析と分割方針を策定した。

## 作業内容

### Issue 精査

6 ファイルの As-Is を実測で検証。ADR-043 の prod/test 分離分析を適用した結果、document.rs（prod ~306行）と approve.rs（prod ~291行）が例外許容基準（prod < 500 + 高凝集）に該当することを発見。

### 責務分析

各ファイルの構造を調査し、責務の混在状況を分析:

| ファイル | 検出された責務数 | 主な混在パターン |
|---------|----------------|----------------|
| Designer.elm (1965行) | 8 | SVG 描画 + Update ロジック + 複数 UI パネル |
| Detail.elm (1479行) | 10 | 承認 + 再提出 + コメント + 複数 View（独立ドメイン混在） |
| instance.rs (prod 754行) | 6 | 状態型定義 + 遷移ロジック + DB 変換 |
| New.elm (1099行) | 4 | Update + View（ADR-043 分析の延長） |

### 分割方針の決定（ADR-062）

- Elm: ページ固有サブモジュール（`Page/X/X/`）パターンを採用。コールバックパターンで View 分離、Update ロジック分離
- Rust: ADR-043 で確立済みの parent.rs + sub-directory パターンを適用
- 例外許容: document.rs、approve.rs（ADR-043 基準適用）

### Story Issue 作成

分割実施用の Story Issue を 4 件作成: #1004, #1005, #1006, #1007

## 判断ログ

| 判断 | 選択 | 理由 |
|------|------|------|
| Elm サブモジュールの配置場所 | Page/X/X/ ディレクトリ | Component/ は再利用可能なもの限定。ページ固有 UI はページ配下に配置 |
| Designer.elm の分割単位 | View 層（4モジュール）+ Update 層 | Nested TEA は ADR-043 で否定済み。サブモジュール分離で十分 |
| Detail.elm の分割軸 | 機能ドメイン（承認/再提出/コメント） | 3 ドメインが状態・メッセージ・UI とも独立しており自然な境界 |
| instance.rs の分割軸 | 責務別（状態型/遷移/DB変換） | handler/workflow.rs と同じパターンで一貫性を維持 |
| document.rs の判定 | 例外許容 | prod 306行（< 500）、6 エンドポイントが全てドキュメント S3 操作で高凝集 |
| approve.rs の判定 | 例外許容 | prod 291行（< 500）、4 関数が全て承認操作で高凝集 |
