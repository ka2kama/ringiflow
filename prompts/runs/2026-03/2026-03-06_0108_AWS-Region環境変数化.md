# AWS SDK クライアントのリージョン環境変数化

Issue: #1047 / PR: #1059

## 概要

AWS SDK クライアントの 3 箇所でハードコードされていた `ap-northeast-1` リージョンを削除し、`aws-config` のデフォルトプロバイダチェーンに委譲した。環境変数 `AWS_REGION` で制御可能にし、env テンプレート・CI 環境にも追加した。

## 実施内容

### Phase 1: `.region()` ハードコード削除 + env ファイル更新

- `ses.rs`, `s3.rs`, `dynamodb.rs` から `.region(aws_config::Region::new("ap-northeast-1"))` を削除
- `.env.template`, `.env.api-test.template`, `scripts/env/generate.sh` に `AWS_REGION=ap-northeast-1` を追加
- `dynamodb_test.rs`（3 テスト関数）と `audit_log_repository_test.rs`（`setup()` 関数）に `dotenvy::dotenv().ok()` を追加
  - `s3_test.rs` では既に同パターンが存在していたが、DynamoDB テストでは欠落していた
  - ハードコードされた `.region()` がこの不整合を隠蔽していた
- `.github/workflows/ci.yaml` の env セクションに `AWS_REGION: ap-northeast-1` を追加
  - CI 環境には `.env` ファイルがないため、`dotenvy` だけでは不十分

### 設計判断

`.region()` を削除してプロバイダチェーンに委譲する方式を採用。環境変数を読んで `.region()` に渡す方式（二重管理）よりもシンプル。`aws-config` のデフォルト挙動（`AWS_REGION` → AWS config file → EC2/ECS メタデータ）を活用する。

## 判断ログ

- 設計判断: プロバイダチェーン委譲を選択（二重管理の回避）
- DynamoDB テストに `dotenvy::dotenv().ok()` 追加時、`s3_test.rs` の既存パターンに統一

## 議論の経緯

- worktree 環境でのセットアップ時、DB やコンテナの問題に対してワークアラウンドを模索する傾向があった。ユーザーから「`just setup` を実行すれば解決する問題を、頑なに避けるのはなぜか」と指摘を受けた。最もシンプルな解決策を最初に試すべきという教訓
- CI で DynamoDB 統合テストが `MissingSigningRegion` で失敗。`.env` ファイルが CI に存在しないことが原因。`ci.yaml` の env セクションに `AWS_REGION` を追加して解決
