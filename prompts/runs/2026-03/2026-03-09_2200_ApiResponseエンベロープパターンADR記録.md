# ApiResponse エンベロープパターン ADR 記録

- Issue: #1093
- PR: #1094
- 日時: 2026-03-09

## 概要

BFF の全エンドポイントで使用している `ApiResponse<T>` エンベロープパターンの設計判断を ADR-065 に記録した。

## 判断ログ

### 初版の判断（選択肢 A: エンベロープ維持 + PaginatedResponse 統合）

十分な調査なしに以下の根拠で選択肢 A を選んだ:
- 選択肢 B の変更規模を「大規模」と過大評価
- 「将来 `meta` が必要になるかもしれない」という YAGNI に反する根拠
- 業界のベストプラクティスを調査していなかった

### ユーザーの指摘を受けた再調査

3 つの並行調査を実施:

1. **変更規模の実測**: 88 箇所の `ApiResponse::new()` は全て同一パターンで 100% 機械的置換可能。フロントエンドは 1 行変更のみ（PR #1092 で集約済み）
2. **業界のベストプラクティス**: GitHub/Stripe/WordPress API はエンベロープ不使用。「HTTP がエンベロープ」が主流の考え方
3. **RingiFlow のメタデータニーズ**: 要件定義書にトップレベルメタデータの要件なし。エラーは RFC 9457、メタデータは HTTP ヘッダーで対応済み

### 最終判断（選択肢 B: エンベロープ廃止）

エンベロープの正当性がないため廃止を決定。変更は全て機械的で低リスク。

## 学び

- 変更規模は「ファイル数」ではなく「機械的置換の可否」で評価すべき
- 設計判断に「将来の拡張性」を根拠にする場合、具体的なシナリオで検証する（YAGNI チェック）
- ADR を書く前にベストプラクティスの調査を完了させるべき
