# SES 連携と環境設定の実装

- Issue: #879
- PR: #1040
- ブランチ: `feature/879-ses-integration`

## 概要

本番環境向けの AWS SES 連携を有効化した。`SesNotificationSender` は通知基盤（#875）で実装済みであり、今回の作業は DI コンテナへの配線と Terraform リソース定義が中心。

## 変更内容

### Phase 1: SES クライアント初期化と app_builder 配線

- `notification/ses.rs` に `create_ses_client()` 追加
- `notification.rs` に `SesClient` 型エイリアス追加（re-export）
- `app_builder.rs` に "ses" match arm 追加、シグネチャに `ses_client: Option<SesClient>` 追加
- `main.rs` で `NOTIFICATION_BACKEND=ses` のときのみ SES クライアントを生成

### Phase 2: Terraform SES ドメイン検証

- `infra/terraform/modules/ses/` に SES モジュール作成
- `aws_sesv2_email_identity`（ドメイン検証 + DKIM）
- `aws_sesv2_configuration_set`（TLS 必須、レピュテーション追跡）

### Phase 3: テスト手順文書

- `docs/60_手順書/03_運用/02_SESメール送信テスト.md` 作成

## 判断ログ

### SES クライアント生成場所

`main.rs` で条件付き生成し `build_app` に `Option<SesClient>` を渡す設計を採用。理由: `build_app` が同期関数であり、AWS SDK クライアント生成は async のため内部で生成不可。S3 クライアントと同じパターン。

### `SesClient` 型エイリアス

`app_builder.rs` が `aws-sdk-sesv2` crate に直接依存するのを避けるため、infra crate で `pub type SesClient = aws_sdk_sesv2::Client` を定義。S3 の `AwsS3Client`（独自ラッパー型）とは異なり、SES はラッパーが不要なため型エイリアスで対応。

### AWS SDK 設定の共通化を見送り

`aws_config::defaults(...).region(...).load().await` が S3/DynamoDB/SES の 3 箇所に存在するが、各サービスのクライアント生成は微妙に異なる（S3: endpoint + force_path_style、DynamoDB: endpoint + credentials、SES: シンプル）ため、現時点では共通化しない。
