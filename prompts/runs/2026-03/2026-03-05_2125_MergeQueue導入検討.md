# Merge Queue 導入検討

## 概要

Issue #757（Merge Queue 導入と品質保証の再設計）に着手した。ADR-064 を作成し、品質保証の分析と設計を完了した後、GitHub Rulesets API で Merge Queue を有効化しようとしたところ、ユーザー所有リポジトリでは Merge Queue が利用できないことが判明した。全変更を元に戻し、Issue はブロッカー付きでオープンのまま残した。

## 実施内容

1. Issue #757 の精査と As-Is 検証
   - Rulesets API で現行設定を確認（`strict_required_status_checks_policy: true`、required checks: CI Success / Claude Auto Review / Claude Rules Check）
   - CI ワークフロー、Claude Auto Review / Rules Check のトリガー構造を確認

2. ADR-064 作成
   - 現行の再レビューが提供する保証の分析（AI レビューは PR diff をレビューするため rebase 後も結果同一）
   - 3 つの選択肢を比較し、選択肢 1（Merge Queue + CI のみ）を採用
   - CI ワークフロー変更、Ruleset 変更、マージコマンドの設計詳細を記載

3. 実装計画（`prompts/plans/757_merge-queue.md`）の作成
   - Phase 1: ADR（完了）
   - Phase 2: CI への `merge_group` トリガー追加
   - Phase 3: Merge Queue 有効化とドキュメント更新

4. Merge Queue 有効化の試行と失敗
   - Rulesets API で `merge_queue` ルールを追加しようとしたところ HTTP 422 エラー
   - `status_check_configuration` パラメータを除外しても `Invalid rule 'merge_queue'` エラー
   - 原因: Merge Queue はユーザー所有リポジトリでは利用不可（Organization 所有が必要）

5. 全変更のリバート
   - CI ワークフローへの `merge_group` トリガー追加を取り消し
   - Rulesets の `strict_required_status_checks_policy` を `true` に復元
   - ADR-064 のステータスを「保留」に変更

## 判断ログ

- Merge Queue が利用不可と判明した時点で、`strict_required_status_checks_policy: false` のみの変更（Merge Queue なし）を検討した。しかし、Issue の本質は品質保証の維持であり、strict checks を無効化するだけでは merge commit での CI 検証がなくなるため品質が低下する。ユーザーの判断により全変更をリバートした
- ADR-064 は設計分析として価値があるため、ステータスを「保留」にして保持。リポジトリの Organization 移行後に再開予定

## 議論の経緯

- Merge Queue が利用不可と判明した際、strict checks の無効化のみの変更を提案した。ユーザーから「品質担保の話でしょう？」と指摘を受け、Issue の Want が throughput 改善ではなく品質保証の再設計であることを再確認した
- ユーザーの判断で全変更をリバートし、Merge Queue が利用可能になるまで Issue をオープンのまま残す方針に決定した
