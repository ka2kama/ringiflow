## 概要

`frontend/src/Page/Workflow/New.elm`（1099行）を 4 つのサブモジュールに分割した。ADR-062 の対象 4。

## 実施内容

### 分割結果

| ファイル | 行数 | 内容 |
|---------|------|------|
| `New.elm` | 221 | オーケストレーター（init, update ルーティング, view レイアウト） |
| `New/Types.elm` | 149 | 共有型定義（Model, PageState, FormState, EditingState, Msg, initEditing） |
| `New/Update.elm` | 540 | updateLoaded, updateEditing, バリデーション, dirty 管理, API 呼び出し |

### 呼び出し側の変更

- `Main.elm`: `NewTypes.Model` / `NewTypes.Msg` で型を参照（Detail.elm 分割と同じパターン）
- `NewTest.elm`: `Page.Workflow.New.Types` から型を直接 import

## 判断ログ

### 3→4 ファイル構成

ADR-062 は当初 3 ファイルを計画していたが、Elm の循環依存制約により Types.elm を追加した 4 ファイル構成とした。Designer.elm（6→7）、Detail.elm（5→6）と同じパターン。Elm のモジュール分割では Types.elm パターンが事実上必須。

### Update.elm の 540行超過

Update.elm は 540行で 500行閾値を 40行超過。`updateEditing` が単一の大きな case 式（Msg のパターンマッチ）であり、case の腕を別モジュールに分離すると責務境界が不明確になる。Designer/Update.elm（696行）と同様の判断で現状維持。

### RemoteData.Loading との名前衝突

`Types.elm` で `PageState(Loading, ...)` を `exposing (..)` で公開し、同時に `RemoteData exposing (RemoteData(..))` を import すると `Loading` が曖昧になる。`RemoteData exposing (RemoteData)` として `RemoteData.Success` / `RemoteData.Failure` のように修飾して回避した。
