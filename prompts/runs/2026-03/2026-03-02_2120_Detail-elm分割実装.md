# Detail.elm 分割実装

Issue: #1005
PR: #1016
ブランチ: `feature/1005-split-detail-elm`

## 概要

`Page/Workflow/Detail.elm`（1479行）を6つのサブモジュールに分割した。Designer.elm 分割（PR #1013）で確立した Types.elm パターンを踏襲し、循環依存を回避しながらドメインごとに update + view を共配置する構成を採用した。

## 実施内容

### 分割結果

| モジュール | 行数 | 責務 |
|-----------|-----:|------|
| Detail.elm | 484 | オーケストレーター（init, routing, layout） |
| Detail/Types.elm | 178 | 共有型定義（Model, Msg, LoadedState 等） |
| Detail/Approval.elm | 300 | 承認/却下/差し戻し操作 + 確認ダイアログ |
| Detail/Resubmit.elm | 456 | 再提出編集モード + フォーム + 承認者選択 |
| Detail/Comments.elm | 167 | コメントスレッド表示 + 投稿 |
| Detail/StepProgress.elm | 106 | ステップ進捗バー（純粋 View） |

### 作業の流れ

1. Types.elm を作成し、共有型定義を抽出
2. Main.elm を更新（`DetailTypes` からの型インポート）
3. `PageState(..)` の `Loading` と `RemoteData(..)` の `Loading` が名前衝突 → `Types.Loading` 等の修飾名で解消
4. StepProgress.elm → Comments.elm → Approval.elm → Resubmit.elm の順にサブモジュールを作成
5. Detail.elm を最終形（オーケストレーター）に書き換え
6. elm-review で未使用インポート 7 件を検出・修正

### コンテキスト復元

セッション途中でコンテキストが切り詰められた。サブモジュール 4 つの作成完了後、Detail.elm の最終書き換え直前で復元し、作業を継続した。

## 判断ログ

- Issue 精査: 元の Issue は「5ファイル」としていたが、Elm の循環依存制約により Types.elm が必要なため「6ファイル」に修正して続行
- 設計判断: Designer.elm 分割では Update.elm に全 update ロジックを集約したが、Detail では各サブモジュールに update + view を共配置した。理由: Detail の各ドメイン（承認、再提出、コメント）は Designer よりも独立性が高い
- Refactor: elm-review の自動検出に従い未使用インポートを除去（7件）。分割時に元の import をそのままコピーしたことが原因
