# 通知バックエンドDIの型安全リファクタリング

- 日時: 2026-03-05
- Issue: #1046
- PR: #1048

## 概要

`build_app` が `Option<SesClient>` を受け取り `.expect()` で unwrap していたパターンを、`main.rs`（コンポジションルート）で `Arc<dyn NotificationSender>` を構築して渡す設計にリファクタリングした。

## 変更内容

### 変更ファイル

| ファイル | 変更内容 |
|---------|---------|
| `backend/apps/core-service/src/main.rs` | 通知バックエンド選択ロジックを追加（SMTP/SES/Noop の match） |
| `backend/apps/core-service/src/app_builder.rs` | シグネチャ変更（`Option<SesClient>` → `Arc<dyn NotificationSender>`）、バックエンド選択ロジック削除 |
| `backend/crates/infra/src/notification.rs` | 不要になった `SesClient` 型エイリアスを削除 |

### 設計判断

通知バックエンド選択ロジックの配置先として `main.rs` を選択。理由:

- S3 クライアント（`Arc<dyn S3Client>`）と同じ DI パターンとの一貫性
- `build_app` の責務を「組み立て済みの依存を受け取ってルーターを構築」に限定（SRP）
- `build_app` が `aws-sdk-sesv2` の具象型に依存しなくなる（DIP）

ADR は不要と判断（既存パターンへの統一であり、新しい設計判断ではない）。
