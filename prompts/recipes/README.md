# prompts/recipes/

Claude Code が使う操作パターンを「手動でも再現可能な形」で記録する。

## 目的

Claude Code がなくても同じ操作ができるようになること。

- Claude Code の暗黙知を形式知に変換する
- 操作の「なぜ」と「どうやって」を両方記録する
- 定着したパターンは `docs/04_手順書/` へ昇格させる

## `docs/04_手順書/` との違い

| | `docs/04_手順書/` | `prompts/recipes/` |
|--|------------------|-------------------|
| 対象 | 定型作業（Issue作成、PR作成など） | 探索的・問題解決的な操作 |
| 形式 | 正式な手順書 | カジュアルなTips集 |
| 安定性 | 確立されたフロー | 発見したパターン（改善の余地あり） |

## ディレクトリ構造

カテゴリ別に整理する。

```
recipes/
├── README.md
├── search/         # 検索・調査系
├── debug/          # デバッグ・トラブルシューティング
├── refactor/       # リファクタリング
└── git/            # Git 操作
```

カテゴリは必要に応じて追加する。

## レシピの書き方

### ファイル名

```
<動詞>-<対象>.md
```

例:
- `search/find-related-files.md`
- `debug/trace-compile-error.md`

### 構成

```markdown
# タイトル（〇〇する）

## いつ使うか

このパターンが有効な状況を箇条書きで。

## 手順

### Claude Code でやる場合

Claude Code への指示例、または使用するツール。

### 手動でやる場合

コマンドや操作手順。コピペで実行できる形式で。

## なぜこの方法か

この手順を選んだ理由、他の方法との比較。
```

## 運用ルール

### いつ記録するか

- Claude Code が非自明な操作で問題を解決したとき
- 同じパターンを2回以上使ったとき
- 「これは知っておくと便利」と思ったとき

### いつ昇格させるか

以下の条件を満たしたら `docs/04_手順書/` に移動:

- 3回以上使用して、手順が安定している
- プロジェクト固有ではなく、汎用的なパターンである
- 正式な手順書として文書化する価値がある
