# prompts/

AI セッション固有の知識を「再現可能な設計資産」として残し、改善・再利用できる形で蓄積する。

開発プロセス全般の知識（改善記録、診断レポート）は [`process/`](../process/) に格納する。

→ 経緯: [ADR-050](../docs/05_ADR/050_プロセス知識ディレクトリの分離.md)

## 何を置くか

- セッションログ（実施記録）→ [`prompts/runs/`](runs/)
- 操作パターン集（再現可能な手順）→ [`prompts/recipes/`](recipes/)
- 計画ファイル（設計の思考過程）→ [`prompts/plans/`](plans/)
- 定番の制約（出力形式、禁止事項、品質基準、テスト方針）
- few-shot（代表例：良い例/悪い例、要約版）

## 置かない（置くなら必ず加工する）

- 秘密情報（鍵・トークン・社内URL・個人情報）
- 生ログの丸ごと保存

保存が必要な場合は、必ず以下を行ってください。
- 伏字・匿名化・要約（「汎用テンプレ + 匿名化した例」に落とす）
- 何が入力で、何が出力かを切り分ける（再利用できる単位にする）

## 推奨運用

- セッションログは [`prompts/runs/`](runs/) に置く
  - 命名: `YYYY-MM-DD_NN_<トピック>.md`
  - 例: `prompts/runs/2026-01-16_07_Issue駆動開発導入.md`
- PR/Issue にセッションログへのリンクを含める

## 文体ガイドライン

runs に保存するプロンプトは、会話のやり取りそのままではなく、第三者が閲覧可能なフォーマルな日本語に整えること。

### 基本原則

- 敬体（です・ます調）または丁寧な依頼形（〜してください）を使用する
- 口語的な表現は避け、文語的な表現を選ぶ
- 誤字脱字、表記揺れを修正する

### 具体例

| 項目 | 修正前（口語） | 修正後（文語） |
|------|--------------|--------------|
| 動詞 | 〜して | 〜してください |
| 疑問 | 〜どうでしょうか | 〜どのような評価になりますか |
| 指示語 | この、あの | 本、当該 |
| 語彙 | つきあわせる | 照合する |
| 語彙 | 作り | 構成、構造 |
| 語彙 | いい感じに | 適切に |
| 委任 | 任せます | お任せします、ご判断ください |

### プロンプト記載例

```
# 修正前
ULIDとUUID v7の比較

# 修正後
ULID と UUID v7 を比較してください。
```

```
# 修正前
CLAUDE.mdの作りを全体的に見直して ultrathink

# 修正後
CLAUDE.md の構成を全体的に見直してください。 ultrathink
```

## セッション引き継ぎ

別セッションに作業を引き継ぐ場合、専用ファイルは不要。以下で十分:

```
Project Board を確認して、次のタスクを進めてください。
```

Project Board: https://github.com/users/ka2kama/projects/1

AI は CLAUDE.md と Issue を読んで必要な情報を取得できる。
