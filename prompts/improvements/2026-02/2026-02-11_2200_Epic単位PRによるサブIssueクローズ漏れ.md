# Epic 単位 PR によるサブ Issue クローズ漏れ

## 事象

PR #426 で Epic #403（Phase 2-2）全体を一つの PR として実装・マージした。PR 本文に `Closes #403` のみ記載したため、サブ Issue（#427〜#432）が全て OPEN のまま残った。

同様のパターンは過去にも発生しており、再発している。

## 原因分析

### 表層的原因

PR 本文にサブ Issue の `Closes` 参照が含まれていなかった。

### 根本原因

Epic 全体を一つの PR で実装するブランチ戦略が、以下の問題を構造的に生んでいた:

1. **レビュータイミングの遅延**: 全 Story 完了後に一括レビュー
2. **PR の巨大化**: レビュー品質の低下
3. **Issue クローズの不自然さ**: `Closes` で 6 件のサブ Issue を列挙する必要があり、忘れやすい

手順書の Epic/Story 運用フローは「Story ごとに実装 → クローズ → 全 Story 完了で Epic クローズ」を想定しているが、実際は 1 Epic = 1 PR で運用されていた。手順とプラクティスの乖離がプロセスギャップとなっていた。

## 対策

サブ Issue のクローズ漏れ自体への対症的対策（PR 本文にサブ Issue を列挙するルール追加）ではなく、根本原因であるブランチ戦略を見直す。

Story-per-PR（各 Story を個別 PR で main にマージ）を標準とすることで:

- 各 PR が 1 Story を `Closes` で自然にクローズ
- レビュー粒度が小さくなり、フィードバックが早期化
- PR サイズが適切になり、レビュー品質が向上

恒久対策: Issue #452
