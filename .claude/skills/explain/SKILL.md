# 解説ドキュメント生成

対象を分析し、E2E フローを中心とした解説ドキュメントを生成する。

## 原則

1. **E2E フローが主役**: ユーザー操作から始まり、レスポンスで終わる sequenceDiagram が中心
2. **コンポーネント間の連携を解説する**: API 契約、型変換、エラー伝播など、1つのファイルからは見えない境界部分
3. **内部処理は書かない**: ファイル内部のロジックはコードが自己説明的であるべき。解説が必要ならコードを改善する

## 引数

$ARGUMENTS

## モード判定

引数が省略された場合は、何を解説するかユーザーに質問する。

| 引数 | モード | 分析対象 |
|------|--------|---------|
| PR 番号（例: `123`） | diff | PR の差分 |
| ブランチ名（例: `feature/34-user-auth`） | diff | ブランチの差分 |
| 機能領域名（例: `認証フロー`） | feature | 既存コード |
| ファイルパス（例: `Page/Workflow/Detail.elm`） | feature | 指定ファイル周辺のコード |
| 省略 | — | ユーザーに質問する |

## 出力

1つの解説ドキュメントを生成する（旧形式の「機能解説 + コード解説」ペアは廃止）。

配置先: `docs/90_実装解説/` の機能ドメインディレクトリ

### 配置の判定

| 対象の性質 | ディレクトリ | 例 |
|-----------|------------|-----|
| ユーザー操作を起点とする機能 | 機能ドメイン名 | `ワークフロー/`、`認証/`、`ファイル管理/` |
| 複数の機能にまたがる基盤 | `横断的関心事/` | RLS、Observability、通知 |

## 手順

### Step 1: 対象の情報を収集

#### diff モード

```bash
# PR 番号指定の場合
gh pr view <PR番号> --json title,body,baseRefName,headRefName,files
gh pr diff <PR番号>

# 現在のブランチの場合
git log --oneline --reverse HEAD --not main
git diff --stat main...HEAD
git diff main...HEAD
```

| 情報 | コマンド | 用途 |
|------|---------|------|
| コミット履歴 | `git log --oneline --reverse HEAD --not main` | 変更の時系列 |
| 変更ファイル統計 | `git diff --stat main...HEAD` | 影響範囲の把握 |
| 差分全体 | `git diff main...HEAD` | 詳細分析 |
| 関連 Issue | PR 本文から抽出 | コンテキスト |

**E2E フローの追跡**: 変更されたファイルから、ユーザー操作 → Elm → BFF → Core Service → DB の流れを追跡する。1つのファイルではなく、リクエストがどのコンポーネントを通過するかを把握する。

#### feature モード

1. 指定された機能領域に関連するファイルを特定する（Glob / Grep）
2. E2E フローを追跡する: Handler → UseCase → Repository の呼び出しチェーンを辿る
3. コンポーネント間の境界（API エンドポイント、型変換）を特定する
4. 関連する設計書、要件定義書を確認する

### Step 2: 出力先を決定

1. `docs/90_実装解説/` の既存ディレクトリを確認する
2. 機能ドメインに該当するディレクトリがあればそこに配置、なければ新規作成する

ファイル命名:

```
docs/90_実装解説/<機能ドメイン>/NN_<トピック>.md
```

- `NN` はディレクトリ内の次の連番
- 命名規則の詳細: [`docs/90_実装解説/README.md`](../../../docs/90_実装解説/README.md#ファイル命名)

出力先をユーザーに提示し、確認を得てから Step 3 に進む。

### Step 3: 解説ドキュメントを生成

対象の性質に応じて、機能ドメインまたは横断的関心事のテンプレートを使用する。

---

#### 機能ドメインのテンプレート

##### 3.1 概要

```markdown
# <機能名>

<!-- diff モード -->
対応 PR: #<PR番号>
対応 Issue: #<Issue番号>

<!-- feature モード -->
対象: <機能領域の説明>

## 概要

<ユーザー視点で「何ができるか」を 1-2 文で記述>
```

##### 3.2 E2E フロー

**文書の中心セクション。** ユーザー操作を起点に、システム全体でどう処理が連携するかを sequenceDiagram で可視化する。

```markdown
## E2E フロー

### 正常系

<Mermaid sequenceDiagram: ユーザー操作からレスポンスまでの全体フロー>

participant の選択基準:

| 対象 | participant |
|------|------------|
| 常に含める | ユーザー、Elm Frontend、BFF、Core Service、PostgreSQL |
| 該当する場合 | Auth Service、Redis、S3、DynamoDB |

### 準正常系（該当する場合）

<バリデーションエラー、権限不足等のフロー>

### 異常系（該当する場合）

<システムエラー時のフロー>
```

sequenceDiagram の粒度:
- **含めるもの**: コンポーネント間のリクエスト/レスポンス、主要なデータ変換、エラー分岐
- **含めないもの**: コンポーネント内部のメソッド呼び出し、ローカル変数の操作

##### 3.3 コンポーネント間の境界

```markdown
## コンポーネント間の境界

### API 契約

| エンドポイント | メソッド | リクエスト型 | レスポンス型 |
|--------------|---------|------------|------------|
| `/api/v1/...` | POST | `CreateRequest` | `Response` |

### 型変換の流れ

<リクエスト/レスポンスがレイヤーを通過する際の型変換を記述>

```
Elm Encoder → JSON → BFF Handler(RequestBody) → CoreClient(CoreRequest) → Core Handler(Request) → UseCase → Domain Entity
```

### エラー伝播

| エラー | 発生箇所 | 伝播経路 | HTTP Status | ユーザーへの表示 |
|-------|---------|---------|-------------|---------------|
| ... | ... | Core → BFF → Elm | ... | ... |
```

セクション内容の取捨選択:
- API 契約: エンドポイントがある場合は常に含める
- 型変換の流れ: レイヤーをまたぐ型変換がある場合
- エラー伝播: エラーケースがある場合

##### 3.4 状態遷移（該当する場合）

```markdown
## 状態遷移

<Mermaid stateDiagram-v2: エンティティのライフサイクル>
```

##### 3.5 設計判断

仕組み・方式レベルの判断を記載する。コード実装レベルの判断（関数の書き方、パターンの選択等）は書かない。

```markdown
## 設計判断

### N. <〜をどうするか>

<なぜこの判断が必要だったか: 1-2文の文脈説明>

| 案 | <判断軸1> | <判断軸2> | <判断軸3> |
|----|----------|----------|----------|
| **<採用案>（採用）** | ... | ... | ... |
| <代替案1> | ... | ... | ... |

採用理由: <1-2文>
```

##### 3.6 関連ドキュメント

```markdown
## 関連ドキュメント

- [設計書](...)
- [横断的関心事: マルチテナントRLS](../横断的関心事/01_マルチテナントRLS.md)
```

---

#### 横断的関心事のテンプレート

##### 概要

```markdown
# <横断的関心事名>

対応 PR: #<PR番号>

## 概要

<何を担保するか、なぜ必要かを 1-2 文で>
```

##### いつ・どこで発動するか

```markdown
## いつ・どこで発動するか

<具体的な機能フロー内での発動タイミングと位置を示す>

| 発動タイミング | 対象 | 例 |
|--------------|------|-----|
| リクエスト受信時 | 全 API | 認証ミドルウェア |
| DB クエリ実行時 | テナントデータ | RLS ポリシー |

<Mermaid sequenceDiagram: 機能フローの中でどこに介入するか>
```

##### 仕組み

```markdown
## 仕組み

<動作原理を記述。内部コードの詳細ではなく、概念レベルで>
```

##### 設計判断

機能ドメインの設計判断と同じ形式。

---

### Step 4: 確認と仕上げ

1. 生成したドキュメントをユーザーに提示する
2. 図表の正確性、リンクの整合性を確認する
3. コミットするかユーザーに確認する
4. コミットメッセージの例: `Add explanation docs for <機能名>`

## セクションの取捨選択

全セクションを機械的に生成しない。対象の性質に応じて取捨選択する。

| セクション | 含める条件 |
|-----------|-----------|
| E2E フロー | 常に含める（機能ドメインの場合） |
| コンポーネント間の境界 | API エンドポイントやレイヤーをまたぐ型変換がある場合 |
| 状態遷移 | エンティティのライフサイクルがある場合 |
| 設計判断 | 仕組み・方式レベルの選択がある場合 |
| いつ・どこで発動するか | 常に含める（横断的関心事の場合） |

含めないもの:
- コードスニペット（行番号付き）
- 型の classDiagram
- テストとコードフローのマッピング
- マイグレーション SQL
- 依存クレート一覧

## ドキュメント規約

- 図表: Mermaid 記法（[docs.md](../../../.claude/rules/docs.md)）
- ノードラベル: `ID["ラベル"]`（flowchart）、`participant ID as ラベル`（sequenceDiagram）
- sequenceDiagram の記法: [実装解説 README > sequenceDiagram の記法ルール](../../../docs/90_実装解説/README.md#sequencediagram-の記法ルール)
- ファイルリンク: 相対パス
- サニタイズ: [docs.md のドキュメント生成時の自己チェック](../../../.claude/rules/docs.md#ドキュメント生成時の自己チェック)
