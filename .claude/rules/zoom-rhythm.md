# 俯瞰・実装リズム（Zoom In/Out Rhythm）

全体（俯瞰）と局所（実装）を往復する。

## 前提: 設計段階では設計を詰める

- 設計段階: 網羅性、完結性、整合性を確認してから実装に入る
- 実装段階: 設計を前提に実装しつつ、必要に応じて設計に戻る
- 設計段階で収束していない計画は、実装に入ってはならない

### 設計ブラッシュアップループ

To-Be と As-Is のギャップがゼロになるまでループする。

1. 計画の初版を書く
2. ギャップを体系的に洗い出す（[ギャップ発見の観点](#ギャップ発見の観点)を使用）
3. 各ギャップを調査・解決する（コード探索、API 検証、仕様確認）
4. 計画を更新する（設計判断を追加、コードスニペットを修正）
5. ループ記録に追記する
6. 2 に戻る（ギャップゼロになるまで）

#### ギャップ発見の観点

→ ISO 25010 品質特性との対応: [KB: ISO 25010 > ギャップ発見観点とのマッピング](../../docs/80_ナレッジベース/methodology/ISO25010.md#ギャップ発見観点とのマッピング)

**欠陥の発見（マイナス → ゼロ）**:

| 観点 | 探すもの |
|------|---------|
| 未定義 | 参照されているが定義されていないもの |
| 曖昧 | 複数の解釈が可能な記述 |
| 競合・エッジケース | 組み合わせの挙動が未定義のもの |
| 不完全なパス | 状態遷移やエラーパスに漏れがあるもの |
| アーキテクチャ不整合 | レイヤー境界や責務分担の矛盾 |
| 責務の蓄積 | 既存モジュールへの機能追加で、既存の責務と異なる関心事が導入されるもの。SRP（単一責務原則）の蓄積的な違反パターン |
| 既存手段の見落とし | 標準ライブラリや既存パターンに代替があるもの |
| デザイントークン乖離 | [デザインガイドライン](../../docs/40_詳細設計書/13_デザインガイドライン.md)に沿っていない実装 |
| アクセシビリティ欠陥 | WCAG 2.1 AA 違反（セマンティック HTML、ARIA、キーボード操作、コントラスト比） |
| 状態網羅漏れ | RemoteData の全バリアント（NotAsked/Loading/Failure/Success）と空データの一部が、view または Cmd（特に DOM 依存の Ports）で未処理のもの |
| 状態依存フィールド | 特定の状態でのみ有効なフィールドがフラットな型に混在しているもの。型安全ステートマシンで分離すべき（[ADR-054](../../docs/70_ADR/054_型安全ステートマシンパターンの標準化.md)） |
<!-- 改善: process/improvements/2026-03/2026-03-05_2300_異なる用途の構造体共用によるダミー値.md -->
| 文脈依存のダミー値 | 同一の型が異なる用途（一覧/詳細、作成/更新等）で使われ、一方の用途でフィールドにダミー値（`Vec::new()`、`Default::default()`、意味のない固定値等）が必要になるもの。用途ごとに型を分離すべき |
<!-- 改善: Issue #1080 テストリスト作成時のテストケース抜け漏れ防止 -->
| テスト層網羅漏れ | テストリストで全テスト層（ユニット/ハンドラ/API/E2E）が明示されていないもの。「該当なし」と判定した層について、操作パスがその層の検証対象（API 境界、UI 操作等）を通過していないか確認する |
| 操作パス網羅漏れ | 完了基準に対応する操作パス（正常系・準正常系・異常系）がテストリストに反映されていないもの。各操作パスについて、テストケースが正常系のみに偏っていないか（準正常系・異常系・境界値が検討されているか）確認する |
| テスト責任の断絶 | Epic の完了基準に紐づく操作パス（正常系・準正常系・異常系）が、テストピラミッドのいずれかの層で、いずれの Story のテスト責任にも割り当てられていないもの |
| セキュリティ境界の欠落 | 認証・認可パスの新規/変更で設計書との突合が欠けているもの、シークレットのハードコーディング、ユーザー入力の未バリデーション |

**品質の向上（ゼロ → プラス）**:

| 観点 | 問い |
|------|------|
| シンプルさ | 同じ結果をより少ない概念で実現できないか |
| 型の活用 | 不正な状態を型レベルで防げないか |
| 責務の明確さ | 各コンポーネントの責務は単一か |
| ベストプラクティス | その分野の推奨パターンに沿っているか |
| ユーザーフィードバック | 操作に対する視覚的フィードバック（loading、success/error 表示）が適切か |
| 視覚的階層 | 情報の優先度がタイポグラフィ・色・スペーシングで表現されているか |
| ビジュアルバランス | 要素のコントラスト・整列・余白が[デザインガイドライン](../../docs/40_詳細設計書/13_デザインガイドライン.md)の原則に沿っているか |

対象の性質に応じて観点を追加する。

#### 完了基準

1. 上記の観点で体系的に探索してギャップがゼロ
2. 各設計判断に選択肢・理由・トレードオフが記載されている
3. 具体的なコードスニペット（関数シグネチャ、型定義）で挙動が一意に確定する

ループ完了後に[収束確認チェックリスト（設計・計画フェーズ）](#設計計画フェーズ)で成果物の品質を検証する。

#### ループ記録（計画ファイルの必須要素）

「観点」列は[ギャップ発見の観点](#ギャップ発見の観点)に対応する。

```markdown
### ループの記録

| ループ | 検出したギャップ | 観点 | 対応 |
|-------|----------------|------|------|
| 1回目 | [何を検出したか] | [観点] | [計画をどう変更したか] |
```

## 往復のリズム

### 実装中の俯瞰: 設計原則レンズ

TDD Refactor ステップで設計原則レンズを適用する。
→ 具体的な問いとトリガー: [TDD 開発フロー > Refactor](dev-flow-tdd.md#refactor-設計を改善する)

| タイミング | レンズ |
|-----------|--------|
| 毎 Refactor | Simple Design（意図・重複・最小性） |
| モジュール/関数の完成時 | SRP・DIP・型の活用 |
| Phase 完了時 | 全レンズ + [品質チェックリスト](dev-flow-issue.md#62-品質チェックリスト) |

発見があった場合のみ、セッションログの判断ログに記録する。

### その他の作業での俯瞰

| タイミング | 俯瞰の内容 |
|-----------|-----------|
| 一つの変更を完了したとき | この変更は全体と調和しているか？ |
| 予期しない発見時 | 局所的対処でいいか？全体に組み込むべきか？ |
| 区切りのとき | ここまでの成果は全体と調和しているか？ |

## 理想駆動による収束

1. To-Be を先に定義する
2. As-Is を To-Be に近づける
3. 俯瞰してギャップを確認する
4. ギャップがゼロになるまで 2-3 を繰り返す

### To-Be の確定と更新

確定タイミング: 設計・計画は探索完了後、実装はテストを書く前、テストはテストリスト作成時、横断検証は全 Phase 完了後。

更新条件: より良い理想状態の発見、外部からの要件変更、技術的制約の発見、探索での新知見。更新する場合は変更理由を明示してから新しい To-Be で往復を再開する。

**禁止:**

- 往復の中で暗黙で To-Be を変更すること
- 「実装が難しいから」という理由で理想を下げること
- 変更理由を記録せずに To-Be を変えること

## 収束確認のチェックリスト

### 設計・計画フェーズ

| # | 観点 | 理想状態（To-Be） | 確認内容 |
|---|------|------------------|---------|
| 1 | 網羅性 | 探索で発見された全対象が計画に含まれている | 対象リストと探索結果を突合し、差分がゼロ。除外はすべて理由付きで記載 |
| 2 | 曖昧さ排除 | 不確定な記述がゼロ | 「あれば」「必要に応じて」等を検索し、該当ゼロを確認 |
| 3 | 設計判断の完結性 | 全ての差異・バリエーションに判断が記載されている | 探索で見つかった差異と計画の判断を突合し、未決定ゼロ |
| 4 | スコープ境界 | 対象と対象外が両方明記されている | 「対象」「対象外」セクションが存在し、暗黙の除外がない |
| 5 | 技術的前提 | コードに現れない前提が考慮されている | HTML/CSS 仕様、フレームワーク挙動、ツール制約を確認済み |
| 6 | 既存ドキュメント整合 | 既存ドキュメントと矛盾がない | 要件定義書、設計書、ADR と照合し、矛盾ゼロ |

### 実装フェーズ

各 TDD サイクルの Refactor で [設計原則レンズ](dev-flow-tdd.md#設計原則レンズ) の問いに答える。

コミット前に品質チェックリストの「コード品質確認」と「品質向上」がすべて OK であること。Phase 完了後に全体を通して検証する。→ [品質チェックリスト](dev-flow-issue.md#62-品質チェックリスト)

#### 暫定値・ワークアラウンド採用時の確認事項

暫定値やワークアラウンドを計画に記載する際は、その値が通る全パス（ユニットテスト、統合テスト、API テスト、開発サーバー）を列挙し、各パスで問題がないことを確認する。

<!-- 改善: process/improvements/2026-02/2026-02-23_1932_ワークアラウンド導入時のフォローアップIssue作成漏れ.md -->
ワークアラウンドを導入する場合は、クリーンアップ用のフォローアップ Issue を作成し、コミットメッセージまたは PR 本文にフォローアップ Issue 番号を記載する。成果物要件: ワークアラウンドを含むコミットまたは PR に、対応するフォローアップ Issue 番号が記載されていること。

### テストフェーズ

テストの理想状態（To-Be）。具体的な確認手順は [品質チェックリスト](dev-flow-issue.md#62-品質チェックリスト) を参照。

| # | 観点 | 理想状態（To-Be） |
|---|------|------------------|
| 1 | テストリスト突合 | 計画と実装が 1:1 で対応している |
| 2 | カバレッジ | 正常系・異常系・境界値が網羅されている |
| 3 | E2E 視点 | ユーザーが UI から操作を完了できる |
| 4 | TDD 遵守 | Red → Green → Refactor のサイクルを守っている |
| 5 | テスト品質 | 各テストが独立した仕様を表現し、冗長なテストがない |

### 横断検証フェーズ（複数 Phase の PR で全 Phase 完了後に実施）

全 Phase 完了後に、PR 全体を俯瞰して横断検証する。→ [品質チェックリスト](dev-flow-issue.md#62-品質チェックリスト)

## 品質を守る二つの砦

設計 → 最初の砦（ブラッシュアップ）→ 実装（設計原則レンズ）→ 最後の砦（品質ゲート）→ レビュー

| | 最初の砦 | 最後の砦 |
|---|---------|---------|
| タイミング | 設計段階（実装前） | 品質ゲート（実装後） |
| プロセス | [ブラッシュアップループ](#設計ブラッシュアップループ) | [品質ゲート](dev-flow-issue.md#6-品質ゲート) |
| マイナス→ゼロ | ギャップ発見（欠陥の発見） | 整合性チェックリスト |
| ゼロ→プラス | ギャップ発見（品質の向上） | 品質向上の最終確認 |

特定コンテキストの詳細ルール（[api.md](api.md), [repository.md](repository.md), [data-store.md](data-store.md)）は俯瞰時の観点を提供する。

## 計画ファイルへの記載（必須）

記載されていない計画は完成とみなさない。

計画ファイルは3つのセクションで構成する。仕様整理（何を作るか）と設計（どう作るか）の分離により、参照する情報源が異なるフェーズのコンテキストを独立に永続化し、圧縮耐性を高める。

```
コンテキスト           ← セッション再開の起点
① 仕様整理（specify）  ← 要件・仕様（Outside-in: ユーザー視点 → システム境界）
② 設計（plan）         ← 実装方針（Inside-out: システム内部 → 実装構造）
③ ブラッシュアップ状態  ← ループの進行状態（中断耐性）
収束確認               ← 品質ゲート
```

### コンテキスト（ファイル冒頭）

新セッションが計画ファイル 1 つで作業を再開できるよう、以下をファイル冒頭（仕様整理や Phase 定義より前）に記載する。

```markdown
## コンテキスト

### 目的
- Issue: #NNN（あれば）
- Want: [本質的なニーズ（1〜2 文）]
- 完了基準:
  - [完了基準1]
  - [完了基準2]

### ブランチ / PR
- ブランチ: `feature/NNN-xxx`
- PR: #NNN（Draft）

### As-Is（探索結果の要約）
[新セッションが再探索なしに作業を続けるために必要な情報: キーとなるファイルパス、発見したパターン、既存コードの構造]

### 進捗
- [ ] Phase 1: [名前]
- [ ] Phase 2: [名前]
```

Issue に紐付かない場合（プロセス改善、ドキュメント整備等）は `Issue:` 行を省略する。Want と完了基準は常に必要。

各項目の記載基準:

| 項目 | 記載すること | 記載しないこと |
|------|------------|--------------|
| Issue | Issue 番号。紐付かない場合は省略 | — |
| Want | 本質的なニーズ | 実装手段（How） |
| 完了基準 | Issue があればその完了基準をインラインでコピー。なければ自分で定義 | Issue 番号への参照のみ |
| ブランチ/PR | ブランチ名と PR 番号。未作成なら「未作成」 | — |
| As-Is | 重要なファイルパス・パターン・制約の要約 | Phase 確認事項と重複する詳細 |
| 進捗 | Phase の完了状態。完了したら `[x]` に更新 | TDD サイクル内のステップ詳細 |

**禁止:**
- Issue がある場合に、完了基準を Issue 番号への参照のみで済ませること
- コンテキストセクションのない計画ファイルで実装を開始すること

### ① 仕様整理（specify）

「何を作るか」を明確にする。要件定義書・機能仕様書・ユーザー操作を参照し、Issue の完了基準をシステムの振る舞いレベルに精緻化する。

仕様整理の成果物は設計フェーズの入力となり、コンテキスト圧縮後も計画ファイルから再読み込みできる。

```markdown
## 仕様整理

### スコープ
- 対象: [変更対象の列挙]
- 対象外: [明示的に除外するもの]

### 操作パス

| # | 操作パス | 分類 | テスト層 |
|---|---------|------|---------|
| 1 | [ユーザーの操作を記述] | 正常系 | E2E |
| 2 | [エラー処理の操作を記述] | 準正常系 | API |
```

→ 操作パスの列挙手順と分類: [TDD 開発フロー > 操作パスの列挙](dev-flow-tdd.md#操作パスの列挙)

操作パスは Issue の完了基準から導出する。完了基準の各項目に対して正常系・準正常系・異常系を検討し、テスト層を割り当てる。

操作パスが存在しない場合は「操作パス: 該当なし（ドメインロジックのみ / 操作パスが存在しない）」と理由付きで明示する。

### ② 設計（plan）

「どう作るか」を決める。コードベース・型定義・既存パターン・ADR を参照し、仕様を実装に落とし込む。

設計判断がある場合は Phase 定義の前に記載する。Phase 定義には確認事項とテストリストを含める。

#### 設計判断

選択肢がある場合に記載する。各判断に選択肢・選定理由・状態（確定 / 仮置き）を記載する。

```markdown
## 設計

### 設計判断

| # | 判断 | 選択肢 | 選定理由 | 状態 |
|---|------|--------|---------|------|
| 1 | [判断内容] | A: xxx / B: yyy | [理由] | 確定 |
| 2 | [判断内容] | A: xxx / B: yyy | 調査中 | 仮置き |
```

#### Phase 定義

Phase 分割基準: 依存順（下位 → 上位レイヤー）。各 Phase に確認事項とテストリストを含める。

##### 確認事項（各 Phase）

実装前に確認すべき型・パターン・ライブラリ API・リファクタリング操作の影響をゲート条件として記載する。
→ 原則: [実装前の確認](pre-implementation.md)

確認事項は Phase の実装に入る前提条件を宣言するもの。Phase 開始時に Read/Grep で実施する。計画ファイルへの状態更新（チェックボックス等）は不要。

```markdown
#### 確認事項
- 型: [確認する型] → [ファイルパス]
- パターン: [参照するパターン] → [参照ファイル]
- ライブラリ: [使用する API] → [Grep 既存使用 or docs.rs URL]
- リファクタリング: [分割対象の重複パターン] → [対象ファイル]
```

確認事項が不要な Phase は「確認事項: なし（既知のパターンのみ）」と明示する。

##### テストリスト（各 Phase）

テストピラミッドの各層を明記する。該当しない層は「該当なし」と記載し、省略しない。操作パス（仕様整理セクション）から逆算してテストを配置する。

```markdown
#### テストリスト

ユニットテスト:
- [ ] 正常系のテスト

ハンドラテスト（該当なし）
API テスト（該当なし）
E2E テスト（該当なし）
```

テスト層の判断基準: HTTP ハンドラ → ハンドラテスト、公開 API → API テスト、UI 操作 → E2E テスト。

### ③ ブラッシュアップ状態

ブラッシュアップループの結果だけでなく進行状態を記録する。ループ途中でコンテキスト圧縮が発生しても、計画ファイルから「どの観点まで検討済みか」「どの判断が仮置きか」を復元できる。

```markdown
## ブラッシュアップ

### ギャップ発見の観点 進行状態

| 観点 | 状態 | メモ |
|------|------|------|
| 未定義 | 完了 | 全参照を確認、未定義なし |
| 曖昧 | 完了 | Phase 2 のエラー型を明確化 |
| 競合・エッジケース | 進行中 | 並行承認のケースを調査中 |
| 不完全なパス | 未着手 | — |
| ... | ... | ... |

### ループの記録

| ループ | 検出したギャップ | 観点 | 対応 |
|-------|----------------|------|------|
| 1回目 | [何を検出したか] | [観点] | [計画をどう変更したか] |

### 未解決の問い
- [調査途中の疑問、保留中の判断]
```

記載要件:
- 観点進行状態: [ギャップ発見の観点](#ギャップ発見の観点)の各項目について「完了 / 進行中 / 未着手」を記録する。全項目が「完了」になったらループ終了
- ループの記録: ループが 1 回以上記録されていること
- 未解決の問い: 仮置きの設計判断、調査途中の疑問を記録する。空の場合は「なし」と明記

#### 永続化タイミング

| 状況 | アクション |
|------|-----------|
| ループ 1 回完了時 | ループの記録を追記、観点進行状態を更新 |
| 大量のコード探索（Read/Grep 10 回以上）を行った | 観点進行状態と未解決の問いを書き出す |
| 設計判断が仮置きになった | 設計判断テーブルの状態を「仮置き」に更新、未解決の問いに追記 |
| 新しい観点の検討に入る前 | 前の観点の結果を書き出す |

## 収束確認（設計・計画）

| # | 観点 | 理想状態（To-Be） | 判定 | 確認内容 |

### セッション境界のガイドライン

設計が重い Issue では、仕様整理完了・設計完了を自然なセッション境界として活用できる。

| セッション境界 | 条件 | 保証されること |
|--------------|------|--------------|
| 仕様整理完了後 | スコープと操作パスが確定 | 次セッションで仕様を再探索する必要がない |
| 設計完了後（推奨） | 収束確認が OK、設計コミット済み | 次セッションは計画ファイルから TDD 実装を開始できる |
| Phase 完了後 | Phase の進捗を `[x]` に更新済み | 次セッションは次の Phase から再開できる |

設計完了後のセッション境界が最も効果的。理由: 仕様と設計の両方が永続化されており、実装セッションはコードベースの探索に集中できる。

## PR 本文への記載（必須）

PR 本文の **品質確認セクション** に収束確認結果を記載する。記載なしで Ready for Review にしない。

品質チェックリスト（6.2）のカテゴリ別に確認内容 1 行で記載する。該当しないカテゴリは `N/A（理由）`。全カテゴリが記載され、各行にコロン以降の確認内容があること。

**実装の場合:**
```markdown
## 品質確認

- 設計・ドキュメント: 詳細設計書 `docs/40_詳細設計書/XX_xxx.md` 更新済み、ADR該当なし、セッションログ `prompts/runs/2026-02/xxx.md`
- Issue との整合: 完了基準・実装計画すべて達成、Want/To-Be 突合で過不足なし
- テスト: テストリスト全項目実装、全層カバー
- コード品質（マイナス→ゼロ）: レイヤー違反なし、既存パターン準拠、OpenAPI同期確認済み
- 品質向上（ゼロ→プラス）: 不要な中間層なし、責務単一
- UI/UX: N/A（バックエンドのみ）
- 横断検証: N/A（単一Phase）
- `just check` pass + CI pass: 全テスト通過
```

注: 「設計・ドキュメント」カテゴリでは、作成・更新したドキュメントの具体的なファイルパスを記載する。「作成済み」「更新済み」のみの記載は検証不能なため不可。該当なしの場合は `ADR該当なし` のように明示する。

**ドキュメント修正の場合:**
```markdown
## 品質確認

- 参照の網羅性: 関連ファイルの参照をすべて更新
- 用語の統一: 新旧用語が混在していないことを確認
- 意図の明確さ: 変更の目的が読み手に伝わることを確認
- `just check` pass + CI pass: 全テスト通過
```

## AI エージェントへの指示

- 計画ファイルは3層構造（仕様整理 → 設計 → ブラッシュアップ状態）で作成する
- 仕様整理と設計を分離し、それぞれの成果物を計画ファイルに永続化する
- ブラッシュアップループ中は、観点進行状態・設計判断の状態・未解決の問いを逐次更新する
- 実装中は TDD Refactor の設計原則レンズで俯瞰する。具体的な問いに答える
- 発見があった場合のみ、セッションログの判断ログに記録する
- TDD 以外の作業では、変更完了時・発見時・区切りで視点を上げる
- 成果物の提示・コミット前に収束を確認する
- 計画ファイルには収束確認セクションを必ず記載する
- PR 本文の品質確認セクションに収束確認結果を記載する
- 往復の実施を逐一報告する必要はない

**禁止:**

- 視点を上げずに実装だけを続けること
- 収束確認を実施せずに成果物を提示・コミットすること
- 収束確認で指摘が見つかったにもかかわらず修正せずに提示すること
- 収束確認セクションのない計画ファイルを提示すること
- 品質確認セクションが空または不十分な状態で品質ゲートを通過済みとすること
- ブラッシュアップ状態セクションのない計画でブラッシュアップループを開始すること
