# 問題解決のアプローチ

問題や指摘を受けたとき、すぐに修正案を出さない。まず腰を据えて考える。

## 適用場面

このフレームワークは以下の場面で適用する:

- 問題や指摘を受けたとき
- Issue に着手するとき — Issue 自体を「仮説」として検証する
- 改善記録の対策を実行するとき — 対策案も「仮説」として検証する
- 設計判断を行うとき

Issue も改善記録の対策も正（source of truth）ではない。書かれた前提・スコープ・完了基準を、このフレームワークで検証してから実装に入る。

→ Issue の精査: [Issue 駆動開発 > 既存 Issue の精査](../../docs/04_手順書/04_開発フロー/01_Issue駆動開発.md#既存-issue-の精査)
→ 対策の性質: [改善記録 > 対策セクションの性質](../../prompts/improvements/README.md#対策セクションの性質)

## 本質: Want と How

```
Want（本質）: ユーザーが意図しない AI の挙動を防ぎたい
  └── How（手段）: 具体的な手順・仕組みを確立する
```

- Want は抽象的・普遍的。How は具体的・領域固有。
- How にこだわりすぎると、別の領域で同じ問題が再発する。
- 常に「この対策は Want を満たすか？」を検証する。

### Want の設定基準

Want はプロジェクト理念レベルに置く。手段（How）を Want に昇格させない。

このプロジェクトの Want は [プロジェクト理念](../../CLAUDE.md#プロジェクト理念) に集約される:

- **品質の追求**: ISO/IEC 25010 に基づくプロダクト品質の体系的追求
- **学習効果の最大化**: 設計判断の理由と代替案を理解すること

以下はすべて How であり、Want ではない:

| How（手段） | それが叶える Want |
|------------|------------------|
| 完了基準を客観的・検証可能にする | 品質の追求 |
| 形式知化する | 品質の追求、学習効果 |
| テストを書く | 品質の追求 |
| ドキュメントを整備する | 品質の追求、学習効果 |

手段が目的化すると、Want に寄与しない活動を正当化するリスクがある。

## フレームワーク

以下の順序で考える:

1. **Want（本質）**: ユーザーが本当に望んでいることは何か？
2. **To-Be（理想状態）**: どうあるべきだったか？
3. **As-Is（現状）**: 実際に何が起きたか？
4. **ギャップ分析**: なぜ理想と現実にずれが生じたか？
5. **根本原因**: 表層的な原因の奥にある構造的な問題は何か？
6. **対策設計**: 根本原因を解消し、Want を満たす対策は何か？

## 禁止事項

- フレームワークを経ずに対策を提示すること
- 「とりあえず」の修正を繰り返すこと
- 指摘を受けてから考え始めること（最初から深く考える）

## 自己チェック

対策を提示する前に自問する:

- この対策は表層的ではないか？
- 理想状態を定義したか？
- 根本原因まで掘り下げたか？

## 既知手法との関連

このフレームワークは実践から帰納的に構築されたもので、以下の既知手法と構造を共有する。

| フレームワークの要素 | 対応する既知手法 |
|---------------------|-----------------|
| To-Be → As-Is → ギャップ分析 | ギャップ分析（Gap Analysis） |
| 全体構造（Want → 根本原因 → 対策） | A3 Thinking（トヨタ生産方式） |
| 根本原因の追究 | なぜなぜ分析（5 Whys） |
| Want（Position の奥にある本質） | Interest（Fisher & Ury『Getting to Yes』） |

独自の追加要素:
- Want をプロジェクト理念レベルに固定し、手段（How）との区別を明示的に管理
- 複数の既知手法を統合した単一フレームワーク（ギャップ分析 + 5 Whys + Interest）
