# ドキュメントライフサイクル

ドキュメントには 2 つのライフサイクル特性がある。特性に応じて AI の振る舞いを変える。

## ライフサイクル分類

| 特性 | 定義 | AI の振る舞い |
|------|------|-------------|
| Living | コードベース・プロジェクトの現在の状態を反映する。古くなると誤解を招く | 関連コード変更時にメンテナンスする |
| Record | ある時点の事実・判断・作業を記録する。古くなっても歴史的価値は変わらない | 内容を書き換えない |

## ディレクトリ別の分類

### Living（メンテナンス対象）

| ディレクトリ | メンテナンスのトリガー |
|---|---|
| `docs/10_要件定義書/` | 要件変更時 |
| `docs/20_機能仕様書/` | 機能変更時 |
| `docs/30_基本設計書/` | アーキテクチャ変更時 |
| `docs/40_詳細設計書/` | 実装変更時 |
| `docs/50_テスト/` | テスト方針変更時 |
| `docs/60_手順書/` | ツール・フロー変更時 |
| `docs/80_ナレッジベース/` | 知識の更新・訂正時 |
| `prompts/recipes/` | パターンの改良時 |

### Record（記録）

| ディレクトリ | 記録する時点 |
|---|---|
| `docs/70_ADR/` | 意思決定時 |
| `docs/90_実装解説/` | PR マージ時 |
| `prompts/runs/` | セッション終了時 |
| `prompts/plans/` | 実装完了後（実装中は更新する） |
| `process/improvements/` | 問題発見時 |
| `process/investigations/` | 調査完了時 |
| `process/reports/` | レポート作成時 |

## Record に対する許容される変更

Record の内容は書き換えないが、以下の変更は許容する。

| 変更種別 | 例 | 理由 |
|---------|---|------|
| リンク・参照パスの修正 | ファイル移動によるリンク切れの修正 | 参照可能性の維持 |
| typo 修正 | 誤字脱字の修正（意味を変えない範囲） | 可読性の維持 |
| 設計された追記ポイント | ADR のステータス変更（廃止/置換）、改善記録の検証セクション追記 | 各ディレクトリの運用ルールで定義済み |

許容されない変更:

- 判断・結論の書き換え
- 事実関係の修正（当時の認識として正しければそのまま残す）
- 新しい知見に基づく内容の追記（→ 新しいドキュメントを作成する）

## 判定フロー

ドキュメントを変更しようとする前に:

1. 対象ディレクトリの分類（Living / Record）を上記の表で確認する
2. Living → 通常通り編集する
3. Record → 変更が「許容される変更」に該当するか確認する
   - 該当する → 変更する
   - 該当しない → 変更しない。新しい知見は別のドキュメント（新しい ADR、新しいナレッジベース記事等）に記録する
