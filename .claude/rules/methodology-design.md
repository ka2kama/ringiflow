# 方法論・プロセス設計の方針

> [ベストプラクティス起点](../../CLAUDE.md#共通アプローチ-ベストプラクティス起点) の方法論・プロセス設計領域での具体化。

このプロジェクトでは、開発プロセスや問題解決フレームワークを設計・改訂する際にも、既知手法を地図として活用する。

## 背景と目的

[最新ベストプラクティス採用方針](latest-practices.md) は技術・ツール領域を対象としているが、方法論・プロセス設計には明示的に適用されていなかった。実際、problem-solving.md や zoom-rhythm.md は複数の既知手法の要素を実践から帰納的に再発明したものだったが、既知手法との照合が行われないまま運用されていた。

既知手法を知ることで:

1. 学習効果の最大化: 先人の知見にアクセスでき、「なぜ効くのか」の理解が深まる
2. 検索可能性の向上: 既知の名前を知ることで、関連文献や研究に到達しやすくなる
3. 品質の底上げ: 既知手法が持つ網羅性や構造を参照できる

## 方針

### 1. 新しいプロセスや方法論を作るとき

既知手法を地図として把握した上で設計する。ただし、地図通りに歩く必要はない。

| ステップ | 内容 |
|---------|------|
| 1. 調査 | 解決したい問題に対応する既知のフレームワークを調べる |
| 2. 理解 | 既知手法の構造・強み・弱みを把握する（地図を読む） |
| 3. 設計 | 既知手法を参照しつつ、実践からの知見も取り入れて設計する |
| 4. 記録 | 既知手法との関連と、独自の追加・調整の理由をドキュメントに記載する |

注意: 既知手法に寄せすぎると、実践から帰納的に生まれる独自の洞察を失う。ringiflow の独自フレームワークは、複数の既知手法を横断的に統合し、プロジェクト固有の痛みから独自の要素を加えたからこそ現在の形に到達した。演繹（既知手法 → 適用）と帰納（実践 → パターン発見）の両方を尊重する。

### 2. 既存のルールを改訂するとき

対応する既知手法がないか確認する。発見した場合は「既知手法との関連」セクションに追記する。

### 3. 独自用語を使うとき

独自用語の使用自体は問題ない。ただし、対応する既知概念がある場合は、初出時に併記する。

例: 「Want」という独自用語に対して、交渉学の Interest（Fisher & Ury）が最も近い概念であることを記録する。

## latest-practices.md との関係

| 項目 | latest-practices.md | methodology-design.md（本ファイル） |
|------|--------------------|------------------------------------|
| 対象 | 技術・ツール（言語、ライブラリ、API、仕様） | 方法論・プロセス（問題解決、開発フロー、品質管理） |
| 参照 | 最新の stable バージョン・推奨 API | 既知のフレームワーク・学術研究 |
| 調整 | プロジェクトの技術スタックに合わせて | プロジェクトの文脈・規模に合わせて |
| 記録 | ADR、コミットメッセージ | ルールファイルの「既知手法との関連」セクション、ナレッジベース |

## AI エージェントへの指示

プロセスやフレームワークを設計・改訂する際:

1. まず既知のフレームワークや学術的な手法を調査し、地図を把握する
2. 地図を参照しつつ、実践からの知見も活かして設計する
3. 既知手法との関連と、独自の追加・調整の理由をドキュメントに記載する
4. 独自用語に対応する既知概念がある場合は併記する

注意: 既知手法を「知っている」ことと「従う」ことは違う。地図は現在地と方角を知るためのもので、地図通りに歩くことが目的ではない。ringiflow の独自フレームワークは、実践から帰納的に複数の既知手法を横断統合し、プロジェクト固有の要素を加えたからこそ現在の形に到達した。この帰納的プロセスの価値を損なわないこと。
