# ルール・スキルファイルの文体規約

ルール・スキルファイルは「実行書」。読者は AI エージェント（実行者）。

## 文体

- 指示的・簡潔に書く
- 解説調（「〜である。なぜなら〜」）を避ける
- 語りかけ調（「良いプログラミングは〜」）を避ける
- 冗長な禁止事項の列挙を避ける（既に明確なルールの言い換えを重ねない）

## 含めるべき内容

| 種類 | 例 |
|------|-----|
| 指示 | 「〜すること」「〜しないこと」 |
| 条件・判断基準 | テーブル、フローチャート |
| 例外 | 「以下の場合のみ許容」 |
| 具体例 | コードスニペット |

## 含めるべきでない内容

以下は本文に含めず、適切な場所に記録する。

| パターン | 記録先 |
|---------|--------|
| 変更経緯（「従来は X だったが Y に変更した」） | コミットメッセージ、改善記録 |
| 背景・目的の散文説明 | ナレッジベース、ADR |
| 旧方式との対比テーブル | 不要（採用方式だけ記載） |
| 理由セクション（「## 理由」で禁止事項を正当化） | 改善記録、ADR |

例外: 「改善の経緯: [リンク](...)」形式の 1 行リンクはトレーサビリティとして許容。

## 具体例

### NG: 背景・理由を散文で説明

```markdown
## 理由

lint ツールはコード品質を保つための重要なガードレールである。
無効化による回避は:
1. 問題の先送り: 根本的な問題を隠蔽し、技術的負債を蓄積する
2. 品質の低下: プロジェクト理念「品質の追求」に反する
```

### OK: 指示だけを記述

```markdown
## 禁止事項

- lint の警告を無効化（`#[allow(...)]`, `{- nolint -}` 等）で回避すること
```

### NG: 旧方式との対比を含む

```markdown
| 言語 | 古いスタイル | 新しいスタイル（採用） |
|------|-------------|----------------------|
| Rust | `mod.rs` でモジュール定義 | ファイル名でモジュール定義 |
```

### OK: 採用方式だけを記載

```markdown
| 言語 | 採用スタイル |
|------|-------------|
| Rust | ファイル名でモジュール定義（Rust 2018+） |
```

### NG: 変更経緯を本文に埋め込む

```markdown
この規約は 2026-02-01 に発生した問題を受けて策定された。
当時、計画段階で洗い出しが不十分だったため、手戻りが発生した。
この経験から、以下のチェックリストを導入する:
```

### OK: 経緯はリンクに留め、指示を記述

```markdown
以下のチェックリストを実施すること:

改善の経緯: [計画段階での網羅性不足](...)
```
